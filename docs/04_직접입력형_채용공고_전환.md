# 직접 입력형 채용공고 전환

## 배경

기존 Step 1은 사용자가 채용공고 URL을 입력하면 백엔드가 URL을 분석해 회사를 추정하는 방식이었다.

```text
채용공고 URL 입력
-> URL 도메인/문자열 기반 회사 추정
-> 회사 직무 목록 반환
-> 사용자가 직무 선택
```

하지만 실제 서비스 관점에서는 URL 기반 방식에 한계가 크다.

- 채용 사이트마다 URL 구조가 다르다.
- 채용공고 본문은 SPA, 로그인, 봇 차단 때문에 안정적으로 수집하기 어렵다.
- 단순 URL만으로 직무명, 담당업무, 자격요건, 우대사항을 정확히 알기 어렵다.
- 기업 alias가 없으면 지원 기업 DB에 있어도 “현재 지원하지 않는 기업입니다”로 떨어질 수 있다.

따라서 MVP에서는 URL 자동 수집보다 사용자가 채용공고 핵심 내용을 직접 입력하는 방식이 안정적이다.

## 변경 방향

Step 1을 다음 입력 항목으로 변경했다.

| 입력 항목 | 설명 |
|---|---|
| 회사명 | 예: 삼성전자 |
| 직무명 | 예: 백엔드 개발자 |
| 담당업무 | 공고의 담당업무 내용 |
| 자격요건 | 공고의 자격요건 내용 |
| 우대사항 | 공고의 우대사항 내용, 선택 입력 |

변경 후 흐름은 다음과 같다.

```text
회사명/직무명/담당업무/자격요건/우대사항 입력
-> 백엔드에 직접 입력 채용공고 저장
-> 회사명을 사전 구축 Company DB와 매칭
-> 직무명을 사전 구축 Job DB와 매칭
-> 분석 기준 직무 후보 반환
-> 사용자가 기준 직무 선택
-> 자소서 입력
-> 면접 유형 선택
-> 분석 생성
```

## 새 API

### `POST /api/job-postings/manual/`

사용자가 직접 입력한 채용공고를 저장하고, 사전 구축 기업/직무 DB와 연결한다.

요청 예시는 다음과 같다.

```json
{
  "company_name": "삼성전자",
  "job_title": "백엔드 개발자",
  "responsibilities": "대규모 트래픽을 처리하는 서버 API 개발",
  "requirements": "Python, 데이터베이스, API 설계 경험",
  "preferred_qualifications": "분산 시스템 경험"
}
```

성공 응답 예시는 다음과 같다.

```json
{
  "supported": true,
  "company": {
    "id": 1,
    "company_name": "삼성전자",
    "industry": "반도체/전자",
    "size": "large"
  },
  "job_posting": {
    "id": 4,
    "company_name": "삼성전자",
    "job_title": "백엔드 개발자",
    "responsibilities": "대규모 트래픽을 처리하는 서버 API 개발",
    "requirements": "Python, 데이터베이스, API 설계 경험",
    "preferred_qualifications": "분산 시스템 경험",
    "resolved": true
  },
  "matched_job": {
    "id": 1,
    "job_title": "신입 백엔드 엔지니어 트랙 00001"
  },
  "jobs": [],
  "jobs_meta": {
    "count": 10,
    "page": 1,
    "page_size": 5
  }
}
```

회사명을 DB에서 찾지 못하면 입력 데이터는 저장하되 `supported=false`로 응답한다.

## DB 저장 구조

직접 입력 데이터는 `job_postings` 테이블에 저장된다.

| 필드 | 저장 내용 |
|---|---|
| `user_id` | 입력한 사용자 |
| `company_id` | 매칭된 회사, 실패하면 null |
| `source_url` | URL 방식 호환용, 직접 입력이면 빈 문자열 |
| `company_name` | 사용자가 입력한 회사명 |
| `job_title` | 사용자가 입력한 직무명 |
| `responsibilities` | 담당업무 |
| `requirements` | 자격요건 |
| `preferred_qualifications` | 우대사항 |
| `raw_text` | 위 입력값을 합친 분석용 공고 본문 |
| `resolved` | 회사 매칭 성공 여부 |
| `created_at` | 저장 시각 |

예시 저장 형태는 다음과 같다.

```text
company_name: 삼성전자
job_title: 백엔드 개발자
responsibilities: 대규모 트래픽을 처리하는 서버 API 개발
requirements: Python, 데이터베이스, API 설계 경험
preferred_qualifications: 분산 시스템 경험
resolved: true
company: 삼성전자
```

## 분석 API와의 연결

직접 입력 API는 분석에 바로 사용할 `Job` 후보를 반환한다.

프론트는 사용자가 선택한 `Job.id`를 기존 분석 API에 전달한다.

```http
POST /api/analyze/
```

요청 예시는 다음과 같다.

```json
{
  "job_id": 1,
  "job_posting_url": "",
  "job_posting_text": "회사명: 삼성전자\n\n직무명: 백엔드 개발자\n\n담당업무:\n...",
  "submitted_cover_letter": "자기소개서 내용",
  "selected_interview_types": ["technical"]
}
```

`job_posting_url`은 더 이상 필수가 아니다. 직접 입력 방식에서는 빈 문자열로 저장된다.

분석 payload에는 다음 데이터가 함께 들어간다.

- 사용자가 직접 입력한 채용공고 본문
- 선택된 사전 구축 직무의 `required_skills`
- 선택된 사전 구축 직무의 `recommended_study_areas`
- 선택된 사전 구축 직무의 `job_description`
- 기업의 산업, 인재상, 조직문화
- 사용자 프로필과 자기소개서

즉, 직접 입력 공고와 사전 구축 직무 DB가 함께 LLM 분석에 사용된다.

## 직무 매칭 방식

직무명은 사용자가 입력한 표현과 사전 구축 데이터의 표현이 다를 수 있다.

예를 들어 사용자는 “백엔드 개발자”라고 입력하지만 DB에는 “백엔드 엔지니어”로 저장되어 있다.

이를 보완하기 위해 키워드 기반 매칭을 적용했다.

| 사용자 입력 예 | 매칭 키워드 |
|---|---|
| 백엔드 개발자, 서버 개발자 | 백엔드 |
| 프론트엔드 개발자 | 프론트엔드 |
| 데이터 분석가, 데이터 엔지니어 | 데이터 |
| 머신러닝, ML | 머신러닝 |
| AI, LLM | AI |
| 클라우드 | 클라우드 |
| DevOps, 데브옵스 | DevOps |
| 보안 | 보안 |
| 임베디드, 펌웨어 | 임베디드 |
| 반도체 공정 | 공정 |
| 배터리 | 배터리 |
| 품질 | 품질 |

매칭된 후보는 `jobs`로 반환되고, 가장 앞의 후보는 `matched_job`으로 제공된다.

## 프론트 변경

`frontend/src/components/analyze/StepJobUrl.vue`는 파일명은 유지하되 내부 UI를 직접 입력 폼으로 변경했다.

변경 전:

```text
채용공고 URL 입력
직접 공고 내용 입력 체크박스
```

변경 후:

```text
회사명 입력
직무명 입력
담당업무 입력
자격요건 입력
우대사항 입력
기업/직무 DB 연결 버튼
분석 기준 직무 선택
```

상위 페이지의 진행 단계명도 `URL 입력`에서 `공고 입력`으로 변경했다.

## 기대 동작

사용자는 이제 URL 없이 다음 방식으로 분석을 시작할 수 있다.

1. `/analyze/new` 접속
2. 회사명 입력
3. 직무명 입력
4. 담당업무 입력
5. 자격요건 입력
6. 우대사항 입력
7. `기업/직무 DB 연결` 클릭
8. 매칭된 기업과 직무 후보 확인
9. 분석 기준 직무 선택
10. 다음 단계로 이동

이후 자소서와 면접 유형을 입력하면 기존 분석 API가 그대로 동작한다.

## 향후 개선

현재는 키워드 기반 직무 매칭이다. 이후에는 다음 방식으로 개선할 수 있다.

- 직무명/담당업무/자격요건을 함께 사용한 점수 기반 매칭
- `Skill` 별도 테이블 도입 후 역량 기반 매칭
- 임베딩 기반 유사도 검색
- 사용자가 매칭 후보를 수정하면 학습 데이터로 저장
- 미지원 회사 입력 시 관리자 검수 큐로 전송
