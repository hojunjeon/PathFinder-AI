# 채용공고 및 자기소개서 DB 저장 흐름

> 현재 GraphRAG 기업 KG 전환 후 자기소개서는 `profiles.cover_letters`가 아니라 분석 단위 `cover_letters` 테이블에 저장한다. 이 문서의 `Profile.cover_letters` 저장 흐름은 과거 구현 기록이며, 현재 기준은 `docs/10_GraphRAG_기업_KG_구현.md`를 따른다.

## 권장 브랜치

이번 작업은 직접 입력한 채용공고와 자기소개서를 DB에 저장하고 분석 흐름에 연결하는 기능 추가다.

권장 브랜치명은 다음과 같다.

```bash
feat/save-manual-posting-cover-letter
```

권장 커밋 메시지는 다음과 같다.

```bash
feat: save manual posting and cover letter inputs
```

## 문제 상황

기존 직접 입력형 Step 1에서는 채용공고를 입력하고 기업/직무 DB 연결을 해도 다음 단계 이동이 불안정할 수 있었다.

주요 원인은 다음과 같다.

- 직무 후보가 없을 때 다음 단계 버튼이 계속 비활성화된다.
- 선택된 직무 id와 실제 직무 목록의 id 타입이 다를 경우 선택 직무를 찾지 못할 수 있다.
- 자기소개서 입력 단계는 하나의 긴 텍스트만 받았고, 자기소개서 항목/답변 구조로 DB에 저장하지 않았다.

## 해결 방향

입력 흐름을 다음처럼 정리했다.

```text
Step 1. 채용공고 직접 입력
-> 회사명, 직무명, 담당업무, 자격요건, 우대사항 입력
-> POST /api/job-postings/manual/
-> job_postings 테이블 저장
-> Company/Job DB 매칭
-> 분석 기준 직무 선택

Step 2. 자기소개서 입력
-> 자기소개서 항목, 답변 입력
-> PUT /api/profile/
-> Profile.cover_letters에 저장
-> 분석 요청 submitted_cover_letter에도 텍스트로 전달
```

## Step 1 채용공고 저장

프론트 입력 항목은 다음과 같다.

| 항목 | 예시 |
|---|---|
| 회사명 | 삼성전자 |
| 직무명 | 백엔드 개발자 |
| 담당업무 | 대규모 트래픽 API 개발 |
| 자격요건 | Python, Database |
| 우대사항 | 분산 시스템 경험 |

백엔드 요청은 다음 API로 전송된다.

```http
POST /api/job-postings/manual/?page_size=30
```

요청 예시는 다음과 같다.

```json
{
  "company_name": "삼성전자",
  "job_title": "백엔드 개발자",
  "responsibilities": "대규모 트래픽 API 개발",
  "requirements": "Python, Database",
  "preferred_qualifications": "분산 시스템 경험"
}
```

저장 테이블은 `job_postings`다.

| 필드 | 저장 내용 |
|---|---|
| `company_name` | 사용자가 입력한 회사명 |
| `job_title` | 사용자가 입력한 직무명 |
| `responsibilities` | 담당업무 |
| `requirements` | 자격요건 |
| `preferred_qualifications` | 우대사항 |
| `raw_text` | 위 입력값을 분석용 텍스트로 합친 값 |
| `company_id` | 매칭된 Company |
| `resolved` | 회사 매칭 성공 여부 |

## Step 1 다음 단계 이동 안정화

다음 보완을 적용했다.

- 직무 후보가 없으면 명확한 오류 메시지 표시
- 매칭된 직무가 있으면 `selectedJobId`를 문자열로 통일
- 다음 버튼 클릭 시 선택 직무를 다시 검증
- 선택 직무를 못 찾으면 다음 단계로 이동하지 않고 오류 표시

이제 채용공고 입력 후 `기업/직무 DB 연결`을 누르고 직무 후보가 표시되면 `다음 단계`로 이동할 수 있다.

## Step 2 자기소개서 저장

기존에는 자기소개서를 하나의 textarea 텍스트로만 받았다.

변경 후에는 여러 개의 자기소개서 항목/답변을 입력할 수 있다.

| 항목 | 설명 |
|---|---|
| `question` | 자기소개서 문항 |
| `answer` | 해당 문항에 대한 답변 |

프론트 입력 예시는 다음과 같다.

```json
[
  {
    "question": "지원동기",
    "answer": "백엔드 시스템 안정화에 기여하고 싶습니다."
  }
]
```

다음 단계 버튼을 누르면 아래 API로 저장된다.

```http
PUT /api/profile/
```

저장 위치는 `profiles.cover_letters` JSON 필드다.

## 분석 요청 연결

자기소개서 항목/답변은 두 가지 방식으로 활용된다.

첫째, DB에는 구조화된 JSON으로 저장된다.

```json
[
  {
    "question": "지원동기",
    "answer": "백엔드 시스템 안정화에 기여하고 싶습니다."
  }
]
```

둘째, 분석 요청에는 LLM이 읽기 쉬운 텍스트로 전달된다.

```text
Q. 지원동기
A. 백엔드 시스템 안정화에 기여하고 싶습니다.
```

분석 API 요청에는 다음 값이 포함된다.

```json
{
  "job_id": 1,
  "job_posting_url": "",
  "job_posting_text": "회사명: 삼성전자\n\n직무명: 백엔드 개발자\n\n담당업무:\n...",
  "submitted_cover_letter": "Q. 지원동기\nA. 백엔드 시스템 안정화에 기여하고 싶습니다.",
  "selected_interview_types": ["technical"]
}
```

## 직무 분석 API와의 연결

직접 입력한 채용공고는 다음 방식으로 앞서 만든 직무 분석 API와 연결된다.

```text
입력 회사명
-> Company DB 매칭

입력 직무명
-> Job DB 키워드 매칭
-> matched_job 반환

사용자가 기준 직무 선택
-> selectedJobId 저장
-> /api/analyze/ 요청의 job_id로 전달

분석 payload
-> 채용공고 직접 입력 본문
-> 사전 구축 Job.required_skills
-> 사전 구축 Job.recommended_study_areas
-> 사전 구축 Job.job_description
-> Profile.cover_letters
-> submitted_cover_letter
```

따라서 사용자가 입력한 실제 공고 내용과 사전 구축된 직무 기준 데이터가 함께 LLM 분석에 사용된다.

## 기대 동작

최종적으로 사용자는 다음 흐름을 수행할 수 있다.

1. 채용공고를 직접 보고 회사명, 직무명, 담당업무, 자격요건, 우대사항 입력
2. 입력한 채용공고가 DB에 저장됨
3. 회사/직무가 사전 구축 DB와 매칭됨
4. 분석 기준 직무 선택 후 다음 단계 이동
5. 자기소개서 항목과 답변 입력
6. 자기소개서가 DB에 저장됨
7. 면접 유형 선택
8. 채용공고, 자기소개서, 사전 구축 직무 DB를 함께 사용해 분석 생성
