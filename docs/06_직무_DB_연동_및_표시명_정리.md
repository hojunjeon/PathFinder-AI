# 직무 DB 연동 및 표시명 정리

> 현재 GraphRAG 기업 KG 전환 후 기존 `jobs` 데이터는 로드맵 분석의 핵심 source-of-truth가 아니다. 직무/역량 확장은 taxonomy와 사용자가 입력한 `job_postings` private evidence를 기준으로 하며, 현재 기준은 `docs/10_GraphRAG_기업_KG_구현.md`를 따른다.

## 1. 결론

현재 채용공고 입력 페이지의 **기업/직무 DB 연결** 버튼은 백엔드 DB의 `Company`, `Job`, `JobPosting` 테이블과 연동되어 동작한다.

다만 데이터 파일은 두 종류로 나뉜다.

| 위치 | 내용 | DB 연동 상태 |
|---|---|---|
| `backend/companies/data/large_company_engineering_jobs.jsonl` | 500개 대기업 메타데이터. `company_name`, `industry`, `size`, `talent_description`, `culture_keywords`만 있음 | `companies.0004_seed_large_company_data` migration으로 `Company`에 적재 |
| `jobs_careers/jobs_careers.jsonl` | 10,000개 직무 데이터. `job_title`, `company_name`, `annual_salary_krw`, `required_experience_years`, `applicant_count` 포함 | `companies.0005_seed_jobs_careers_if_empty` migration으로 `Job`이 비어 있을 때만 적재 |

즉, `backend/companies/data`의 JSONL은 이름과 달리 현재 구조상 “직무 상세”가 아니라 “기업 메타데이터”에 가깝다. 실제 직무 후보 목록은 `Job` 테이블에서 나오며, fresh DB에서는 `jobs_careers/jobs_careers.jsonl`을 기반으로 자동 적재되도록 보완했다.

## 2. 기업/직무 DB 연결 버튼 동작 원리

### 2.1 프론트엔드 입력

파일:

- `frontend/src/components/analyze/StepJobUrl.vue`

사용자가 입력하는 값:

- 회사명
- 직무명
- 담당업무
- 자격요건
- 우대사항

사용자가 **기업/직무 DB 연결** 버튼을 누르면 다음 API를 호출한다.

```http
POST /api/job-postings/manual/?page_size=30
```

전송 payload는 사용자가 입력한 채용공고 정보다.

### 2.2 백엔드 수동 채용공고 저장 및 회사 매칭

파일:

- `backend/companies/views.py`
- `ManualJobPostingView`

백엔드는 다음 순서로 처리한다.

1. `ManualJobPostingSerializer`로 입력값 검증
2. `resolve_company_from_name()`으로 입력 회사명을 DB의 `Company`와 매칭
3. 사용자가 입력한 공고 내용을 `JobPosting` 테이블에 저장
4. 회사가 매칭되지 않으면 `supported: false`와 404 반환
5. 회사가 매칭되면 해당 회사의 `Job` 목록에서 입력 직무명과 가까운 직무를 찾음

### 2.3 직무 후보 매칭

파일:

- `backend/companies/views.py`
- `match_jobs_for_manual_posting()`
- `infer_job_keywords()`

동작 방식:

1. 입력 직무명을 소문자/공백 제거 형태로 정규화
2. 키워드 규칙으로 직무군을 추론
   - 예: `설비`, `보전` → `설비기술`
   - 예: `backend`, `서버`, `백엔드` → `백엔드`
   - 예: `dataengineer`, `ETL` → `데이터 엔지니어`
3. 추론된 키워드로 `Job.job_title__icontains` 검색
4. 검색 결과가 있으면 해당 목록 반환
5. 없으면 해당 회사의 전체 직무 목록 반환

응답에는 다음 값이 포함된다.

```json
{
  "supported": true,
  "company": { "...": "..." },
  "job_posting": { "...": "..." },
  "matched_job": { "id": 851, "job_title": "신입 설비기술 엔지니어" },
  "jobs": [
    { "id": 851, "job_title": "신입 설비기술 엔지니어" }
  ],
  "jobs_meta": { "count": 1, "page": 1, "page_size": 30 }
}
```

프론트엔드는 `jobs` 목록을 `<select>`에 보여주고, 선택된 직무의 `id`를 다음 분석 단계로 넘긴다.

## 3. 분석 생성까지 이어지는 흐름

### 3.1 Step 1 → Step 2

파일:

- `frontend/src/views/AnalyzeCreateView.vue`

`StepJobUrl.vue`가 `next` 이벤트로 다음 값을 넘긴다.

- `company`
- `jobId`
- 선택된 `job`
- 사용자가 입력한 채용공고 본문 `job_posting_text`

### 3.2 최종 분석 요청

면접 유형 선택 후 프론트엔드는 다음 API를 호출한다.

```http
POST /api/analyze/
```

주요 payload:

```json
{
  "job_id": 851,
  "job_posting_url": "",
  "job_posting_text": "회사명: ...\n\n직무명: ...",
  "submitted_cover_letter": "...",
  "selected_interview_types": ["technical"]
}
```

### 3.3 백엔드 분석 생성

파일:

- `backend/analysis/views.py`
- `backend/analysis/services.py`

동작 순서:

1. `job_id`로 `Job` DB 레코드 조회
2. `Analysis`를 `pending` 상태로 생성
3. `build_llm_payload()`에서 사용자 프로필, 회사 정보, 직무 정보, 채용공고 본문을 묶음
4. `call_llm_server()`로 LLM 서버에 전달
5. 결과를 `Analysis`에 저장하고 `done` 상태로 응답

즉, 분석은 프론트에서 보이는 문자열만으로 수행되는 것이 아니라, 선택된 `job_id`를 통해 DB의 `Job` 레코드를 다시 읽고 그 내용을 기반으로 수행된다.

## 4. `설비 엔지니어 0007`처럼 보이던 이유

DB의 원본 `Job.job_title` 일부가 다음처럼 내부 식별자를 포함하고 있었다.

```text
신입 설비기술 엔지니어 트랙 00851
설비 엔지니어 0007
```

이 숫자는 데이터셋 생성 과정에서 직무명을 고유하게 만들기 위한 내부 suffix에 가깝다. 사용자가 선택하거나 LLM이 이해해야 할 직무명에는 불필요하다.

기존에는 `JobSerializer`, `JobSearchSerializer`, `AnalysisResultSerializer`, `build_llm_payload()`가 DB 원본 `job_title`을 그대로 사용했기 때문에 화면과 분석 payload에 suffix가 노출될 수 있었다.

## 5. 반영한 수정사항

### 5.1 표시용 직무명 정규화 함수 추가

파일:

- `backend/companies/job_titles.py`

추가 함수:

```python
display_job_title(job_title)
```

제거 대상:

- 끝의 `트랙 00851`
- 끝의 `0007`
- 즉, 정규식 기준 `(?:트랙\s*)?\d{4,}$`

DB 원본값은 유지하고, 사용자-facing 응답과 LLM payload에서만 정리한다.

### 5.2 직무 API 응답 정리

파일:

- `backend/companies/serializers.py`

수정:

- `JobSerializer.job_title`
- `JobSearchSerializer.job_title`

이제 `/api/job-postings/manual/`, `/api/companies/<id>/jobs/`, `/api/jobs/`가 반환하는 `job_title`에는 내부 숫자 suffix가 보이지 않는다.

### 5.3 분석 payload 정리

파일:

- `backend/analysis/services.py`

수정:

- `payload["job_info"]["직무명"]`에 DB 원본 대신 `display_job_title(job.job_title)` 사용

따라서 LLM 서버에는 `신입 설비기술 엔지니어 트랙 00851`이 아니라 `신입 설비기술 엔지니어`가 전달된다.

### 5.4 분석 결과/히스토리 응답 정리

파일:

- `backend/analysis/serializers.py`

수정:

- `AnalysisResultSerializer.job_title`도 표시용 직무명으로 반환

따라서 분석 결과 화면과 히스토리 화면에도 내부 suffix가 노출되지 않는다.

### 5.5 migration 보완

추가된 migration:

- `backend/companies/migrations/0004_seed_large_company_data.py`
- `backend/companies/migrations/0005_seed_jobs_careers_if_empty.py`

역할:

1. `0004`는 `backend/companies/data/large_company_engineering_jobs.jsonl`을 `Company`에 적재
2. `0005`는 `Job` 테이블이 비어 있는 fresh DB에서만 `jobs_careers/jobs_careers.jsonl`을 `Job`에 적재
3. 이미 운영/로컬 DB에 직무가 있으면 기존 직무 데이터를 덮어쓰지 않고 skip

## 6. 수정 후 기대 동작

사용자가 채용공고 입력 페이지에서 다음처럼 입력한다고 가정한다.

```text
회사명: 삼성전자
직무명: 설비 엔지니어
```

버튼 클릭 후 내부적으로는 DB의 특정 `Job.id`가 선택될 수 있다.

```text
DB 원본: 신입 설비기술 엔지니어 트랙 00851
```

하지만 사용자 화면과 분석 payload에는 다음처럼 표시된다.

```text
신입 설비기술 엔지니어
```

즉, DB 매칭 안정성은 유지하면서 화면/분석에는 자연스러운 직무명만 사용한다.
