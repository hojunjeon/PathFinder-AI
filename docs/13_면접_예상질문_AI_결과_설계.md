# 면접 예상 질문 AI 결과 설계

## 1. 문서 목적

PathFinder AI는 취업 준비생이 지원 기업과 직무에 맞춰 면접을 준비할 수 있도록 다음 기능을 제공하는 것을 목적으로 한다.

- 지원 직무에서 공부해야 할 직무 지식 분석
- 채용공고가 요구하는 역량과 사용자 경험의 연결
- 사용자의 자기소개서와 이력서를 바탕으로 한 개인화 면접 예상 질문 제공
- 예상 질문에 답하기 위해 보완해야 할 지식과 답변 방향 안내

이 문서는 서비스의 핵심 결과인 **면접 예상 질문을 어떤 구조로 생성하고 사용자에게 어떤 형식으로 제공할지** 정의한다.

현재 단계에서는 구현하지 않으며, 본 문서의 설계를 검토한 후 데이터 모델, LLM 프롬프트, API 및 화면 구현을 진행한다.

## 2. 분석에 활용하는 데이터

### 2.1 기업 및 직무 정보

서비스 DB에 사전 구축된 데이터를 사용한다.

- 기업명
- 산업
- 기업 규모
- 인재상
- 조직문화 키워드
- 직무명
- 직무 설명
- 직무 관련 키워드
- 요구 역량
- 우대 역량
- 추천 학습 분야
- 면접 단계 및 면접 유형

### 2.2 자기소개서

사용자가 직접 입력한 항목별 자기소개서 데이터를 사용한다.

```json
[
  {
    "question": "자기소개서 항목",
    "answer": "사용자 답변"
  }
]
```

자기소개서에서는 다음 내용을 추출한다.

- 사용자가 주장하는 강점
- 주요 경험
- 수행한 행동
- 문제 해결 과정
- 협업 및 갈등 경험
- 성과와 정량적 결과
- 지원 동기
- 자기소개서 내용 중 면접에서 추가 검증이 필요한 주장

### 2.3 채용공고

사용자가 직접 입력한 채용공고 데이터를 사용한다.

- 회사명
- 직무명
- 담당 업무
- 요구 역량
- 자격 요건
- 우대사항
- 기타 공고 원문

채용공고는 직무 요구사항을 판단하는 가장 우선적인 근거로 사용한다.

### 2.4 이력서 및 프로필

회원가입 또는 프로필 작성 과정에서 저장한 데이터를 사용한다.

- 전공
- 학교 및 학력
- 자격증
- 프로젝트
- 수상 내역
- 경력
- 프로젝트 기술 스택
- 프로젝트 기간
- 프로젝트 및 경력의 정량적 성과

이력서 데이터는 사용자의 경험을 객관적으로 확인하고 자기소개서 내용과 교차 검증하는 데 활용한다.

## 3. 핵심 설계 원칙

### 3.1 예상 질문을 핵심 결과로 제공한다

현재 서비스는 역량 분석과 학습 로드맵 안에 예상 질문을 포함하고 있다. 개선 구조에서는 예상 질문을 부가 데이터가 아니라 최상위 핵심 결과로 제공한다.

```text
기존
역량 분석 → 학습 로드맵 → 로드맵 내부의 예상 질문

개선
요구 역량 분석 → 예상 질문 생성 → 질문별 학습 항목 도출
```

학습 로드맵은 질문을 생성하기 위한 목적이 아니라, 생성된 질문에 답하기 위해 필요한 학습 내용을 정리한 파생 결과로 취급한다.

### 3.2 모든 질문에 생성 근거를 제공한다

각 질문은 최소 하나 이상의 입력 데이터 근거를 가져야 한다.

- 채용공고의 요구사항
- 기업 및 직무 DB의 요구 역량
- 자기소개서의 특정 답변
- 이력서의 특정 프로젝트, 경력, 자격증 또는 수상

입력 데이터에서 근거를 찾을 수 없는 경험이나 사실을 AI가 임의로 생성해서는 안 된다.

### 3.3 완성형 모범답안보다 답변 방향을 제공한다

서비스가 사용자를 대신하여 완성형 답변을 만들어 주는 것보다, 사용자가 자신의 경험을 바탕으로 답변할 수 있도록 다음 정보를 제공하는 것이 적절하다.

- 면접관의 질문 의도
- 답변 구성 순서
- 반드시 포함할 내용
- 피해야 할 답변 방식
- 관련 꼬리질문
- 공부해야 할 직무 지식

완성형 답변 생성은 사용자가 직접 답변 초안을 작성한 이후의 별도 기능으로 분리할 수 있다.

### 3.4 근거 없는 정량 점수를 피한다

AI가 명확한 산정 공식 없이 생성한 다음과 같은 점수는 신뢰성을 낮출 수 있다.

- 직무 적합도 83%
- 합격 가능성 75%
- 역량 점수 90점

대신 다음과 같은 근거 기반 상태를 제공한다.

- 충분한 근거가 확인됨
- 일부 경험은 있으나 설명 보완 필요
- 관련 경험이 확인되지 않음
- 입력 정보가 부족하여 판단하기 어려움

## 4. 결과 화면의 권장 구조

## 4.1 면접 준비 요약

결과 화면 상단에서 전체 분석 내용을 간결하게 제공한다.

- 지원 기업
- 지원 직무
- 선택한 면접 유형
- 채용공고 핵심 요구 역량
- 활용 가능한 사용자 경험
- 보완이 필요한 직무 지식
- 가장 먼저 준비해야 할 질문 3개
- 입력 데이터 부족 또는 분석 주의사항

예시:

```text
지원 직무: 백엔드 개발자

핵심 요구 역량
- REST API 설계
- 데이터 정합성 관리
- 장애 대응 및 협업

활용 가능한 경험
- 주문 처리 프로젝트
- 팀 프로젝트의 API 설계 경험

우선 보완 영역
- 트랜잭션 격리 수준
- 장애 상황의 정량적 결과 설명
```

## 4.2 예상 질문 목록

질문은 다음 분류를 기준으로 제공한다.

1. 직무 지식 질문
2. 프로젝트 및 경험 심층 질문
3. 자기소개서 기반 질문
4. 채용공고 요구 역량 검증 질문
5. 기업 및 직무 지원동기 질문
6. 인성 및 협업 질문
7. 약점 및 경험 공백 검증 질문

초기 MVP에서는 총 12개 내외의 질문을 제공하는 방안을 권장한다.

| 우선순위 | 권장 개수 | 의미 |
|---|---:|---|
| 필수 준비 | 5개 | 실제 면접에서 검증 가능성이 높고 지원자 경험 또는 공고와 직접 연결된 질문 |
| 권장 준비 | 5개 | 직무 수행 능력과 경험의 깊이를 확인하기 위한 질문 |
| 추가 대비 | 2개 | 기업 특성, 인성, 약점 또는 확장 꼬리질문 |

질문 개수는 입력 데이터의 양과 선택한 면접 유형에 따라 달라질 수 있다. 근거가 부족한 경우 개수를 맞추기 위해 질문을 억지로 생성하지 않는다.

## 4.3 학습할 내용

각 예상 질문에 연결된 `study_topics`를 모아서 학습 목록을 만든다.

- 우선순위
- 학습 개념
- 질문과 연결된 이유
- 공부해야 할 세부 항목
- 예상 학습 시간 또는 권장 일수
- 관련 예상 질문

## 4.4 분석 근거

사용자가 결과를 신뢰할 수 있도록 AI가 어떤 데이터를 사용했는지 보여준다.

- 채용공고에서 추출한 핵심 문장
- 기업 및 직무 DB의 관련 역량
- 자기소개서에서 연결된 항목
- 이력서에서 연결된 프로젝트 또는 경력
- 데이터가 부족하여 판단하지 못한 영역

## 5. 질문 카드 형식

각 질문은 다음 정보를 포함한다.

```text
[필수 준비] 프로젝트 심층 · 기술면접 · 중급

질문
프로젝트에서 데이터 정합성 문제가 발생했을 때
어떤 기준으로 트랜잭션 범위를 결정했나요?

질문 의도
실제 트랜잭션 이해도와 문제 해결 과정을 확인합니다.

선정 이유
- 채용공고에서 데이터 정합성 경험을 요구합니다.
- 자기소개서에서 주문 처리 프로젝트를 언급했습니다.
- 이력서에는 구체적인 장애 해결 과정이 작성되어 있지 않습니다.

답변 구성
1. 당시 문제 상황
2. 본인이 맡은 역할
3. 검토한 대안
4. 선택한 방법과 이유
5. 정량적 결과
6. 회고 및 개선점

반드시 포함할 내용
- 본인이 직접 수행한 부분
- 다른 방법을 선택하지 않은 이유
- 결과를 검증한 방법

피해야 할 답변
- 팀이 수행한 일을 모두 본인의 성과처럼 설명
- 기술 선택의 이유 없이 사용 기술만 나열
- 결과나 검증 방법이 없는 답변

꼬리질문
- 분산 환경이었다면 어떻게 달라지나요?
- 트랜잭션 범위를 줄일 방법은 없었나요?

공부할 내용
- ACID
- 트랜잭션 격리 수준
- 트랜잭션 경계
- 동시성 문제
```

### 질문 카드 필수 필드

| 필드 | 설명 |
|---|---|
| `priority` | 필수, 권장, 추가 대비 |
| `category` | 직무 지식, 프로젝트, 자기소개서, 지원동기, 인성 등 |
| `interview_type` | 기술, 실무, 인성, 컬처핏, PT 등 |
| `difficulty` | 기초, 중급, 심화 |
| `question` | 예상 질문 본문 |
| `evaluation_intent` | 면접관이 질문을 통해 확인하려는 내용 |
| `why_selected` | 이 질문을 선정한 이유 |
| `evidence` | 질문 생성에 사용한 실제 입력 근거 |
| `answer_guide` | 답변 구성과 필수 내용 |
| `follow_up_questions` | 예상 꼬리질문 |
| `study_topics` | 질문에 답하기 위해 공부할 내용 |
| `confidence` | 질문 근거의 충분성 |

## 7. AI 처리 단계

한 번의 프롬프트에서 모든 분석과 질문을 생성하기보다 역할이 다른 단계를 분리하는 것을 권장한다.

## 7.1 1단계: 입력 정규화

LLM 호출 전에 Django 백엔드 또는 별도 전처리 계층에서 데이터를 공통 형식으로 변환한다.

```text
기업 정보
→ 인재상, 문화, 직무 키워드

채용공고
→ 담당 업무, 필수 역량, 우대 역량

자기소개서
→ 경험, 행동, 성과, 주장, 지원동기

이력서
→ 프로젝트, 경력, 기술, 자격증, 정량적 결과
```

각 데이터에는 질문 생성 근거로 다시 참조할 수 있도록 `source_id`를 부여한다.

예시:

```json
{
  "source_id": "resume-project-1",
  "source_type": "resume_project",
  "title": "주문 처리 서비스",
  "facts": [
    "Spring Boot 사용",
    "주문 API 개발",
    "응답 시간 30% 감소"
  ]
}
```

## 7.2 2단계: 요구 역량 및 사용자 경험 매칭

다음 연결 관계를 분석한다.

```text
채용공고 요구 역량
→ 관련 직무 지식
→ 사용자 경험 근거
→ 경험 또는 설명의 부족 영역
→ 면접에서 검증할 주제
```

이 단계의 출력은 다음과 같다.

- 채용공고 핵심 요구 역량
- 기업 및 직무 DB의 보조 요구 역량
- 사용자 강점과 근거
- 설명 보완이 필요한 경험
- 관련 경험이 없는 역량
- 예상 질문 후보 주제
- 데이터 부족 경고

## 7.3 3단계: 개인화 질문 생성

2단계 분석 결과와 DB의 질문 템플릿을 사용하여 질문을 개인화한다.

```text
직무 질문 템플릿
+ 채용공고의 실제 요구사항
+ 사용자의 실제 경험
+ 선택한 면접 유형
→ 개인화 예상 질문
```

예시:

```text
일반 질문 템플릿
"프로젝트에서 기술 A를 선택한 이유는 무엇인가요?"

개인화 질문
"주문 처리 프로젝트에서 Redis를 사용한 이유는 무엇이며,
DB 조회만 사용하는 방식과 비교해 어떤 기준으로 결정했나요?"
```

초기 단계에서는 별도의 파인튜닝보다 **DB 검색 및 근거 제공과 LLM 생성을 결합한 RAG 방식**이 적절하다.

## 7.4 4단계: 질문 검증 및 정렬

LLM 결과를 그대로 사용자에게 전달하지 않고 다음 항목을 검사한다.

- 모든 질문에 입력 근거가 있는가
- 입력 데이터에 없는 경험이나 성과가 추가되지 않았는가
- 비슷한 질문이 중복되지 않았는가
- 선택한 면접 유형과 질문 유형이 일치하는가
- 질문 분류가 한 영역에 지나치게 집중되지 않았는가
- 꼬리질문이 본 질문과 연결되는가
- 학습 항목이 질문에 답하는 데 실제로 필요한가
- 질문 우선순위가 근거의 중요도와 일치하는가

초기 MVP에서는 다음 구조를 권장한다.

```text
LLM 호출 1
요구 역량 및 사용자 경험 분석

LLM 호출 2
개인화 예상 질문 및 답변 가이드 생성

백엔드
JSON 스키마 검증, 근거 검증, 중복 제거, 개수 및 분류 조정
```

질문 품질이 충분하지 않다면 추후 별도의 검증 LLM 단계를 추가한다.

## 8. 질문 우선순위 기준

질문의 우선순위는 다음 요소를 기준으로 결정한다.

| 기준 | 설명 |
|---|---|
| 채용공고 직접성 | 담당 업무 또는 필수 요구사항에 직접 명시되어 있는가 |
| 사용자 경험 연결성 | 자기소개서나 이력서에 관련 경험이 존재하는가 |
| 검증 필요성 | 사용자가 주장했지만 역할, 행동 또는 결과가 불명확한가 |
| 직무 중요도 | 해당 직무에서 핵심적으로 요구되는 지식인가 |
| 기업 특화성 | 기업의 사업, 인재상 또는 조직문화와 연결되는가 |
| 면접 유형 적합성 | 사용자가 선택한 면접 유형에서 실제로 다룰 가능성이 높은가 |

### 우선순위 예시

```text
필수 준비
- 채용공고의 필수 역량과 사용자 프로젝트가 직접 연결됨
- 자기소개서에 작성한 핵심 성과의 검증 질문
- 직무의 필수 지식이지만 사용자의 경험 근거가 부족함

권장 준비
- 우대사항과 관련된 경험
- 프로젝트 선택 이유 또는 대안 비교 질문
- 협업과 문제 해결 방식의 심층 질문

추가 대비
- 기업문화 및 지원동기 질문
- 약점, 실패, 갈등과 관련된 확장 질문
- 기본 질문에서 이어질 수 있는 심화 질문
```

## 9. 신뢰도 표시 기준

질문이 실제 면접에 출제될 확률을 의미하는 수치형 점수 대신, 질문 생성 근거의 충분성을 표시한다.

### High

- 채용공고의 명시적 요구사항과 사용자 경험이 모두 존재
- 자기소개서 또는 이력서의 구체적인 문장을 근거로 사용

### Medium

- 직무상 중요한 역량이지만 사용자 경험의 구체성이 부족
- 기업 및 직무 DB에는 존재하지만 채용공고의 직접 근거는 약함

### Low

- 일반적인 직무 질문으로서 의미는 있으나 개인화 근거가 부족
- 입력 데이터가 부족하여 제한적으로 생성한 질문

신뢰도가 낮은 질문은 필수 준비 항목으로 배치하지 않는다.

## 11. 현재 구현 구조와의 차이

현재 LLM 응답과 분석 결과는 다음 구조를 중심으로 한다.

```json
{
  "competency_gap": {},
  "text_roadmap": "",
  "timeline_data": []
}
```

현재 구조의 한계는 다음과 같다.

- 예상 질문이 `timeline_data`의 하위 항목으로 포함되어 핵심 결과로 드러나지 않는다.
- 질문이 어떤 입력 데이터를 근거로 생성되었는지 명확하지 않다.
- 질문의 우선순위, 면접 유형, 난이도 및 평가 의도가 구조적으로 분리되지 않는다.
- 학습 계획과 예상 질문의 책임이 하나의 구조에 섞여 있다.
- 화면에서 표시하는 일부 역량 점수는 실제 계산 근거가 부족할 수 있다.

개선 구조에서는 다음과 같이 변경한다.

```json
{
  "summary": {},
  "questions": [],
  "study_plan": [],
  "metadata": {}
}
```

- `questions`를 최상위 핵심 데이터로 승격한다.
- 질문마다 입력 근거를 연결한다.
- 학습 계획은 질문의 `study_topics`에서 파생한다.
- 근거 없는 점수 대신 근거 상태와 입력 충족도를 제공한다.

## 12. 구현 전 확정이 필요한 사항

다음 항목은 구현 전에 검토하고 확정해야 한다.

1. 예상 질문 기본 개수를 12개로 설정할지 여부
2. 질문 우선순위를 `필수`, `권장`, `추가 대비`로 구분할지 여부
3. 완성형 모범답안을 기본 결과에서 제외할지 여부
4. 모든 질문에 선정 이유와 입력 근거를 공개할지 여부
5. LLM을 역량 분석과 질문 생성의 2단계로 나눌지 여부
6. 기존 `timeline_data`를 유지하면서 점진적으로 전환할지 여부
7. 질문별 사용자 답변 작성 및 AI 피드백 기능을 후속 범위에 포함할지 여부
8. 직무 질문 템플릿 및 지식 개념 DB를 어느 수준까지 사전 구축할지 여부

## 13. 권장 초기 결정안

현재 서비스 목적과 MVP 범위를 기준으로 다음 방향을 권장한다.

- 예상 질문 기본 개수: 12개 내외
- 필수 준비 5개, 권장 준비 5개, 추가 대비 2개
- 완성형 모범답안 대신 답변 구조와 필수 포인트 제공
- 모든 질문에 선정 이유와 입력 근거 제공
- 역량 분석과 질문 생성을 분리한 LLM 2단계 구성
- JSON 스키마 및 근거 검증은 백엔드에서 수행
- 예상 질문을 핵심 결과로 제공
- 학습 로드맵은 질문에 연결된 지식 항목으로부터 생성
- 근거 없는 적합도 및 합격 가능성 점수는 제공하지 않음
- 입력 정보가 부족하면 질문 수를 억지로 채우지 않고 경고를 제공

이 설계안에 대한 검토가 완료된 이후 프롬프트, API 응답 모델, DB 저장 구조 및 결과 화면의 상세 구현 계획을 작성한다.

## 14. 역량 분석 기능 구현 결과

### 14.1 구현 범위

이번 구현에서는 전체 예상 질문 결과 구조를 한 번에 교체하지 않고, 기존 분석 결과의 `competency_gap` 영역을 우선 개선했다.

구현한 핵심 기능은 다음과 같다.

- 지원자의 실제 경험을 기반으로 한 강점 분석
- 지원 직무 대비 보완이 필요한 역량 분석
- 채용공고에서 확인되는 직무 핵심 역량 정리
- 각 분석 결과에 판단 근거와 면접 준비 행동 제공
- 근거 없는 직무 적합도 점수와 퍼센트 제거
- 기존 문자열 배열 응답과 신규 구조화 응답 동시 지원

기존 `timeline_data` 기반 면접 준비 질문과 체크리스트는 유지했다. 따라서 이번 작업은 전체 예상 질문 API 개편의 선행 단계이며, 역량 분석 결과의 신뢰성과 가독성을 먼저 개선한 작업이다.

## 15. 기능 정의

## 15.1 기능 목적

역량 분석은 사용자가 특정 기업의 직무에 지원할 때 다음 내용을 빠르게 판단할 수 있도록 하는 기능이다.

1. 내 경험 중 면접에서 강점으로 활용할 수 있는 경험은 무엇인가
2. 해당 직무를 준비하기 위해 보완해야 할 지식이나 설명은 무엇인가
3. 채용공고에서 중요하게 보는 핵심 역량은 무엇인가
4. 보완 항목을 해결하기 위해 면접 전에 어떤 행동을 해야 하는가

단순히 기술 키워드를 나열하거나 점수를 제공하는 것이 아니라, **경험과 직무 요구사항 사이의 연결 관계**를 보여주는 것을 목표로 한다.

## 15.2 입력 데이터

분석에는 기존 GMS 요청 데이터 구조를 사용한다.

| 입력 | 활용 방식 |
|---|---|
| 채용공고 | 담당 업무, 필수 요건, 우대사항을 판단하는 최우선 근거 |
| 사용자 프로필 | 전공, 학력, 경력, 프로젝트, 자격증, 수상 내역 확인 |
| 자기소개서 | 사용자의 행동, 역할, 문제 해결 과정, 성과와 주장 확인 |
| 기업 정보 | 산업, 인재상, 조직문화 등 보조 판단 근거 |
| 직무 DB | 요구 역량, 우대 역량, 추천 학습 분야 등 보조 근거 |

## 15.3 출력 항목

### 강점 경험

사용자의 입력에서 실제로 확인되는 경험만 사용한다.

| 필드 | 설명 |
|---|---|
| `keyword` | 2~6어절의 경험 중심 핵심 키워드 |
| `experience` | 강점으로 사용할 프로젝트, 경력, 협업 또는 수상 경험 |
| `evidence` | 프로필 또는 자기소개서에서 확인된 사실 |
| `job_relevance` | 해당 경험이 지원 직무와 연결되는 이유 |
| `interview_focus` | 면접 답변에서 강조해야 할 내용 |

예시:

```json
{
  "keyword": "주문 API 성능 개선",
  "experience": "주문 조회 API 개선 프로젝트",
  "evidence": "응답 시간을 비교한 기록이 있습니다.",
  "job_relevance": "대규모 트래픽 처리 업무와 연결됩니다.",
  "interview_focus": "병목 분석 과정과 본인 역할을 강조합니다."
}
```

### 보완 역량

보완 역량을 모두 동일한 약점으로 표현하지 않고, 원인에 따라 네 가지 유형으로 구분한다.

| 유형 | 의미 |
|---|---|
| `knowledge` | 직무 지식이나 기술 이해가 부족함 |
| `articulation` | 경험은 있지만 역할, 행동, 성과 또는 선택 이유 설명이 부족함 |
| `experience` | 직무가 요구하지만 관련 경험 근거가 확인되지 않음 |
| `insufficient_data` | 입력 정보가 부족해 판단할 수 없음 |

| 필드 | 설명 |
|---|---|
| `keyword` | 보완할 역량의 핵심 키워드 |
| `gap_type` | 보완 유형 |
| `reason` | 보완이 필요하다고 판단한 이유 |
| `evidence` | 채용공고 요구 또는 입력 부족 근거 |
| `action` | 사용자가 면접 전에 수행할 준비 행동 |
| `priority` | `high`, `medium`, `low` 우선순위 |

### 직무 핵심 역량

채용공고와 직무 DB에서 확인되는 요구 역량을 정리한다.

| 필드 | 설명 |
|---|---|
| `keyword` | 직무 핵심 역량 |
| `importance` | 필수 `required` 또는 우대 `preferred` |
| `evidence` | 채용공고 또는 직무 DB의 판단 근거 |

## 17. 구현 과정

## 17.1 기존 데이터 흐름 분석

먼저 다음 데이터 흐름을 확인했다.

```text
Vue 분석 생성 화면
→ Django /api/analyze/
→ FastAPI /llm/roadmap
→ GMS OpenAI Gateway
→ Django JSONField 저장
→ Vue 분석 결과 화면
```

확인 결과 Django의 `competency_gap`은 `JSONField`로 저장되고 있어, 배열 내부의 문자열을 객체로 변경하더라도 DB 마이그레이션은 필요하지 않았다.

## 17.2 GMS 프롬프트 개선

`llm_server/roadmap_prompt.py`의 프롬프트에 다음 규칙을 추가했다.

- 채용공고를 가장 우선적인 판단 근거로 사용
- 프로필과 자기소개서에서 확인되는 경험만 강점으로 사용
- 입력에 없는 경험, 역할, 성과와 수치를 생성하지 않음
- 강점 키워드를 경험 중심의 짧은 명사구로 생성
- 보완 역량을 네 가지 유형으로 분류
- 입력이 부족하면 약점으로 단정하지 않고 `insufficient_data` 사용
- 보완 항목마다 구체적인 준비 행동 생성
- 강점과 보완 역량은 각각 최대 5개로 제한
- 직무 핵심 역량은 최대 8개로 제한
- 점수, 적합도 퍼센트, 합격 가능성 생성 금지

GMS 호출 방식과 사용 모델은 기존 구조를 유지했다.

```text
환경 변수: GMS_KEY
모델: gpt-5-nano
응답 형식: JSON object
```

## 17.3 LLM 응답 정규화

LLM이 항상 완전히 동일한 JSON을 반환한다고 가정할 수 없으므로 `llm_server/main.py`에 정규화 계층을 추가했다.

주요 처리 내용:

- 신규 객체 배열 응답 정규화
- 기존 문자열 배열을 신규 객체 형태로 변환
- 누락 필드에 안전한 기본값 적용
- 잘못된 `gap_type`, `priority`, `importance` 값을 허용 값으로 보정
- 배열이 아닌 값이 반환되면 빈 배열로 처리
- 빈 키워드 항목 제거
- 항목 최대 개수 제한

기존 문자열 응답:

```json

  "required_competencies": ["Python"]
}
```

정규화 이후:

```json
{
  "strengths": [
    {
      "keyword": "프로젝트 경험",
      "experience": "",
      "evidence": "",
      "job_relevance": "",
      "interview_focus": ""
    }
  ],
  "gaps": [
    {
      "keyword": "시스템 설계",
      "gap_type": "knowledge",
      "reason": "",
      "evidence": "",
      "action": "",
      "priority": "medium"
    }
  ]
}
```

이 값은 실제 분석 공식이나 GMS 판단 근거가 없는 임의 값이므로 다음 코드를 제거했다.

- 문자열 기반 결정적 해시 점수 생성
- 강점, 요구, 보완 역량 퍼센트 표시
- 데이터가 없을 때 표시되는 Java, Spring, 알고리즘 등의 고정 fallback 점수
- 결과 사이드바의 직무 매칭도 메뉴

현재는 점수 대신 실제 경험 근거, 보완 이유와 준비 행동을 제공한다.

## 19. 트러블슈팅

## 19.1 기존 분석 결과와 신규 결과의 형식 차이

### 문제

기존 저장 데이터는 문자열 배열이지만 신규 GMS 응답은 객체 배열이다.

신규 UI가 객체 형식만 지원하면 기존 사용자가 저장한 분석 결과를 열 때 화면이 비거나 오류가 발생할 수 있었다.

### 해결

LLM 서버와 프론트엔드 양쪽에 호환 처리를 적용했다.

- LLM 서버는 문자열 항목을 기본값이 포함된 객체로 변환
- 프론트엔드는 문자열과 객체를 모두 렌더링
- 필수 키워드가 비어 있는 항목은 제외

이 방식으로 기존 DB 데이터의 일괄 마이그레이션 없이 점진적으로 신규 구조를 적용했다.

## 19.2 GMS 응답의 비정상 배열 타입

### 문제

LLM이 다음과 같이 배열 대신 `null`, 문자열 또는 객체를 반환하면 반복 처리 과정에서 예외가 발생할 수 있었다.

```json
{
  "strengths": null,
  "gaps": "시스템 설계",
  "required_competencies": {
    "keyword": "Python"
  }
}
```

### 해결

`_as_list()` 정규화 함수를 추가하여 실제 배열만 처리하고, 나머지 타입은 빈 배열로 변경했다.

비정상 응답 때문에 전체 분석 API가 실패하지 않도록 방어 로직과 테스트를 추가했다.

## 19.3 근거 없는 점수의 신뢰성 문제

### 문제

기존 직무 역량 매칭도는 키워드 문자열을 해시 처리하여 일정 범위의 퍼센트를 생성했다.

따라서 동일한 문자열에는 동일한 값이 나오지만, 다음 정보를 반영하지 않았다.

- 채용공고 중요도
- 실제 경험 수준
- 자기소개서의 구체성
- GMS 분석 결과
- 객관적인 평가 기준

사용자에게는 실제 AI 분석 점수처럼 보일 수 있어 잘못된 신뢰를 줄 위험이 있었다.

### 해결

직무 역량 매칭도와 fallback 점수를 완전히 제거했다.

점수 대신 다음 정보를 제공하도록 변경했다.

- 어떤 경험이 강점인지
- 무엇을 근거로 판단했는지
- 직무와 어떻게 연결되는지
- 어떤 역량을 왜 보완해야 하는지
- 면접 전에 무엇을 준비해야 하는지

## 19.4 Windows PowerShell의 npm 실행 제한

### 문제

PowerShell 실행 정책 때문에 다음 명령이 실패했다.

```text
npm.ps1 cannot be loaded because running scripts is disabled
```

### 해결

PowerShell 스크립트인 `npm.ps1` 대신 Windows 명령 실행 파일인 `npm.cmd`와 `npx.cmd`를 사용했다.

```powershell
npm.cmd run build
npm.cmd test
npx.cmd playwright test
```

## 19.5 Python 기본 환경에 pytest가 없음

### 문제

시스템 Python으로 테스트를 실행했을 때 `pytest` 모듈을 찾지 못했다.

### 해결

프로젝트에 이미 구성된 각 서비스의 가상환경 Python을 사용했다.

```powershell
llm_server\venv\Scripts\python.exe -m pytest
backend\venv\Scripts\python.exe -m pytest
```

새 패키지를 설치하지 않고 프로젝트의 기존 실행 환경에서 테스트했다.

## 19.6 E2E 체크박스 선택자 불일치

### 문제

로드맵 화면의 체크박스 접근 가능한 이름은 소제목인 `모션 플래닝`이 아니라 실제 면접 질문 문장이었다.

기존 E2E 테스트가 소제목으로 체크박스를 찾으면서 요소를 찾지 못했다.

### 해결

화면의 실제 접근성 구조를 확인하고 질문 문장을 기준으로 체크박스를 선택하도록 테스트를 수정했다.

```javascript
page.getByLabel('A가 아니라 B 방식을 채택한 이유는 무엇인가요?')
```

## 19.7 전체 회귀 테스트에서 확인된 비관련 실패

### 백엔드

전체 백엔드 테스트 48개 중 46개가 통과했고, 기업 테스트 2개가 실패했다.

실패 원인은 테스트 DB에 이미 존재하는 삼성전자 시드 데이터와 테스트의 생성 가정이 충돌한 것이다.

- 기존 직무가 매칭되어 fallback 직무가 생성되지 않음
- 동일 기업명 생성 시 UNIQUE 제약 조건 충돌

이번 역량 분석 변경 영역인 `analysis/tests/test_analysis.py`의 9개 테스트는 모두 통과했다.

### 프론트엔드

전체 E2E 9개 중 8개가 통과했고 대시보드 테스트 1개가 실패했다.

실패 원인은 다크모드 버튼을 `🌙` 이름으로 찾는 기존 선택자가 현재 UI와 일치하지 않아 타임아웃이 발생한 것이다. 역량 분석 E2E 2개는 모두 통과했다.

해당 실패들은 이번 역량 분석 변경 파일과 직접 관련되지 않아 별도 회귀 개선 대상으로 남겼다.

## 22. 후속 개선 과제

이번 구현은 역량 분석을 구조화한 단계다. 서비스의 최종 목적인 개인화 예상 질문 제공을 강화하려면 다음 작업이 필요하다.

1. `questions`를 최상위 결과 데이터로 분리
2. 질문별 생성 근거와 `source_id` 저장
3. 질문 우선순위, 난이도, 면접 유형과 평가 의도 구조화
4. 질문별 답변 초안 작성과 AI 피드백 기능
5. 역량 분석과 질문 생성을 분리한 2단계 GMS 호출 검토
6. JSON Schema 또는 Pydantic 모델을 활용한 더 엄격한 응답 검증
7. 동일·유사 강점과 보완 키워드의 의미 기반 중복 제거
8. 실제 사용자 평가를 통한 질문 및 역량 분석 품질 측정
9. 전체 백엔드 기업 시드 테스트와 대시보드 E2E 선택자 회귀 수정

현재 구조에서는 역량 분석이 사용자의 경험과 직무 요구사항을 연결하고, 이후 예상 질문과 학습 항목을 생성하기 위한 근거 데이터 역할을 한다.

## 23. 역량 분석 2차 개선 배경

기존 구현은 경험별로 경험 근거, 직무 연결, 답변 포인트를 자세히 제공했다. 정보의 근거를 이해하기에는 유용했지만, 사용자가 결과 화면을 처음 보았을 때 다음 판단을 빠르게 내리기 어려웠다.

- 어떤 역량을 면접에서 적극적으로 어필해야 하는가
- 어떤 역량은 경험은 있지만 답변 정리가 필요한가
- 어떤 역량은 직무에 필요하지만 경험이나 지식이 부족해 학습해야 하는가
- 어떤 역량은 현재 입력 정보만으로 판단하기 어려운가

서비스의 목적은 긴 분석문을 읽게 하는 것이 아니라, 지원자가 자신의 준비 상태를 한눈에 파악하고 바로 면접 준비 행동으로 전환하도록 돕는 것이다. 따라서 역량 분석의 중심을 상세한 경험 설명에서 **핵심 역량 키워드와 준비 상태**로 변경했다.

기존의 상세 경험 근거와 직무 연결 정보는 삭제하는 대신 `준비 항목`으로 이동시켰다. 결과적으로 두 영역의 역할을 다음과 같이 분리했다.

| 영역 | 역할 |
|---|---|
| 역량 분석 | 핵심 역량별 현재 준비 상태를 빠르게 스캔 |
| 준비 항목 | 지식 주제별로 경험 연결, 학습 방향, 예상 질문을 구체적으로 안내 |

## 24. 개선된 역량 분석 기능 정의

### 24.1 사용자에게 보여주는 네 가지 상태

| 상태 | 의미 | 사용자 행동 |
|---|---|---|
| `strength` | 직무와 관련된 경험과 근거가 충분함 | 면접에서 적극적으로 어필 |
| `articulate` | 관련 경험은 있으나 직무 언어와 답변 구조가 부족함 | 경험을 직무 역량 중심으로 재구성 |
| `study` | 직무에는 중요하지만 경험 또는 지식 근거가 부족함 | 우선 학습 후 대응 전략 준비 |
| `insufficient_data` | 현재 입력 정보로 강점 또는 부족 여부를 판단하기 어려움 | 프로필·자기소개서 보완 또는 추가 확인 |

화면에서는 이를 각각 다음 문구로 표시한다.

- 어필 가능
- 답변 정리
- 학습 필요
- 판단 보류

### 24.2 키워드 중심 출력 구조

GMS 응답의 `competency_gap` 내부에 `competency_map`을 추가했다.

```json
{
  "competency_map": [
    {
      "keyword": "로봇 제어",
      "status": "strength",
      "importance": "required",
      "signal": "제어 프로젝트 경험이 직무 요구와 직접 연결됨",
      "action": "제어 성능 개선 과정과 수치 결과를 중심으로 어필"
    },
    {
      "keyword": "모션 플래닝",
      "status": "articulate",
      "importance": "required",
      "signal": "경로 생성 경험은 있으나 직무 용어 연결이 약함",
      "action": "경로 계획 기준과 트레이드오프를 답변 구조로 정리"
    },
    {
      "keyword": "산업용 통신",
      "status": "study",
      "importance": "preferred",
      "signal": "채용공고에는 필요하지만 입력된 경험 근거가 없음",
      "action": "EtherCAT의 동작 원리와 적용 사례를 우선 학습"
    }
  ]
}
```

각 카드에는 긴 설명 대신 다음 정보만 노출한다.

1. 핵심 역량 키워드
2. 필수 또는 우대 여부
3. 현재 상태를 판단한 짧은 신호
4. 바로 실행할 다음 행동

### 24.3 직무 역량 매치도 시각화 정책

직무 역량 매치도는 사용자가 부족한 영역을 시각적으로 찾게 하는 목적은 유지하되, 근거 없는 점수나 백분율은 사용하지 않는다.

현재 구현은 `competency_map`의 상태별 개수를 기준으로 분포 막대를 구성한다.

```text
어필 가능 2개 | 답변 정리 1개 | 학습 필요 2개 | 판단 보류 1개
```

분포 막대는 전체 키워드 중 각 상태가 차지하는 비중만 표현한다. 이는 AI가 임의로 산출한 역량 점수가 아니라, 실제로 분류된 키워드 개수를 시각화한 것이다.

## 25. 준비 항목 정보 구조 설계

### 25.1 준비 항목의 목적

준비 항목은 단순한 할 일 목록이 아니다. 다음 질문에 순서대로 답하도록 설계했다.

1. 이 직무에서 어떤 지식 분야가 중요한가
2. 해당 분야의 세부 지식은 무엇인가
3. 왜 이 내용을 준비해야 하는가
4. 내 프로필이나 자기소개서에 연결할 경험이 있는가
5. 경험이 있다면 어떻게 어필하거나 답변을 정리해야 하는가
6. 경험이 없다면 무엇을 공부하고 어떻게 우회 답변해야 하는가
7. 실제 면접에서는 어떤 질문이 나올 수 있는가

### 25.2 지식 계층 구조

준비 항목은 다음 계층으로 구성한다.

```text
직무 지식 분야
└─ 세부 지식 키워드
   ├─ 준비 유형
   ├─ 채용공고·직무 DB 근거
   ├─ 내 경험 연결
   ├─ 우선 학습 개념
   ├─ 준비 접근법
   └─ 예상 질문
```

유일로보틱스의 로봇 관련 직무를 예로 들면 다음과 같이 구성할 수 있다.

```text
로보틱스
├─ 역기구학
│  ├─ 준비 유형: 어필
│  ├─ 내 경험: 로봇팔 제어 프로젝트
│  └─ 질문: 목표 자세를 관절각으로 변환할 때 어떤 방식을 사용했나요?
└─ 모션 플래닝
   ├─ 준비 유형: 답변 정리
   ├─ 내 경험: 경로 생성 프로젝트 일부
   └─ 질문: 경로 최적화 기준과 충돌 회피 전략을 설명해 주세요.

산업용 통신
└─ EtherCAT
   ├─ 준비 유형: 학습
   ├─ 내 경험: 직접 연결되는 경험 없음
   └─ 질문: 실시간 로봇 제어에서 EtherCAT을 사용하는 이유는 무엇인가요?
```

### 25.3 세 가지 준비 유형

| 준비 유형 | 판단 기준 | 결과 화면의 안내 |
|---|---|---|
| `appeal` | 직무 지식과 직접 연결되는 경험 근거가 있음 | 성과와 역할을 중심으로 적극 어필 |
| `organize` | 유사 경험은 있으나 직접 연결 또는 설명 구조가 약함 | 경험을 직무 용어와 STAR 구조로 재정리 |
| `study` | 직무에는 필요하지만 관련 경험이나 지식이 없음 | 핵심 개념 학습 후 유사 경험·학습 의지로 대응 |

이 분류를 통해 사용자는 각 항목을 보며 다음과 같이 판단할 수 있다.

- “이 경험은 직무와 관련도가 높으니 반드시 어필해야겠다.”
- “관련 경험은 있지만 설명이 약하니 직무 언어로 다시 정리해야겠다.”
- “직무에는 필요한데 모르는 내용이므로 우선 공부해야겠다.”
- “직접 경험은 없지만 유사 경험과 학습 과정을 연결해 답변해야겠다.”

## 26. 준비 항목 생성에 사용하는 정보 기준

준비 항목은 모든 입력 데이터를 같은 비중으로 사용하지 않는다. 각 데이터의 역할을 분리해 생성 기준을 명확하게 했다.

### 26.1 채용공고

가장 우선하는 기준이다.

- 주요 업무에서 실제 수행할 작업 추출
- 요구 역량에서 필수 지식과 기술 추출
- 우대사항에서 차별화 가능한 세부 지식 추출
- 반복되는 명사와 기술 용어를 핵심 키워드 후보로 사용

채용공고는 “무엇을 준비해야 하는가”를 결정한다.

### 26.2 기업·직무 DB

채용공고의 짧거나 모호한 표현을 보완한다.

- 기업이 속한 산업과 제품 특성
- 직무의 일반적인 지식 체계
- 기업·직무 관련 키워드
- 상위 지식 분야와 하위 세부 주제의 분류

예를 들어 채용공고에 `로봇 제어 알고리즘`만 적혀 있더라도 직무 DB를 이용해 `로보틱스 → 역기구학`, `로보틱스 → 모션 플래닝`과 같은 학습 계층을 구성한다.

### 26.3 프로필·이력서

정량적이고 비교적 고정된 경험 근거를 찾는 데 사용한다.

- 전공과 수강 기반 지식
- 자격증
- 프로젝트
- 수상 내역
- 경력
- 기술 스택

프로필은 “실제로 해본 적이 있는가”와 “어떤 경험을 어필할 수 있는가”를 판단한다.

### 26.4 자기소개서

경험의 맥락과 행동 근거를 찾는 데 사용한다.

- 어떤 문제를 해결했는가
- 지원자가 맡은 역할은 무엇인가
- 어떤 방식으로 행동했는가
- 결과와 배운 점은 무엇인가
- 경험을 어떤 직무 역량으로 설명할 수 있는가

자기소개서는 “그 경험을 면접에서 어떻게 말할 것인가”를 결정한다.

### 26.5 최종 판단 흐름

```text
채용공고에서 필수 업무·역량 추출
        ↓
기업·직무 DB로 지식 분야와 세부 키워드 보완
        ↓
프로필·이력서에서 직접 경험 탐색
        ↓
자기소개서에서 행동·성과·설명 근거 탐색
        ↓
직접 경험 충분 → appeal
유사 경험 존재 → organize
경험·지식 근거 없음 → study
판단 자료 부족 → insufficient_data
```

- Django JSONField 저장
- 역량 상태별 키워드 표시
- 지식 분야와 세부 키워드 표시
- 경험 연결과 경험 부족 상태 표시
- 학습 키워드와 준비 전략 표시
- 예상 질문 체크리스트 동작

## 28. 구현 과정에서 발생한 트러블슈팅

### 28.1 기존 분석 결과와 새 UI의 데이터 구조 불일치

#### 문제

기존 저장 결과에는 `competency_map`과 `preparation_type`이 없어 새 UI가 비어 보일 수 있었다.

#### 해결

LLM 서버에서 기존 `strengths`, `gaps`, `required_competencies`를 새 상태 모델로 변환하는 호환 계층을 추가했다. 프론트엔드에도 레거시 결과를 다시 분류하는 fallback을 두어 기존 분석 이력을 계속 열 수 있도록 했다.

### 28.2 경험이 없는 학습 항목에 잘못된 경험이 표시되는 문제

#### 문제

`matched_experience`가 빈 문자열인 경우 기존 `evidence` 필드로 fallback하면서, 실제로는 경험이 없는 `study` 항목에도 경험이 있는 것처럼 표시될 가능성이 있었다.

#### 해결

필드 존재 여부를 `hasOwnProperty`로 확인했다. `matched_experience`가 명시적으로 빈 값이면 빈 값 자체를 유지하고, 필드가 아예 없는 레거시 데이터에서만 기존 근거를 사용하도록 수정했다.

### 28.3 역량 매치도를 점수로 표현할 때의 신뢰성 문제

#### 문제

GMS가 생성한 70점, 80% 같은 값은 명확한 산식이 없으면 사용자가 실제 평가 점수로 오해할 수 있다.

#### 해결

점수 생성과 표시를 제거했다. 대신 실제로 분류된 역량 키워드의 상태별 개수를 이용한 분포 막대로 변경했다. 시각적 비교 기능은 유지하면서 근거 없는 정밀도를 피했다.

### 28.4 상세 정보가 역량 분석의 가독성을 떨어뜨리는 문제

#### 문제

경험 근거, 직무 연결, 답변 포인트를 한 카드에 모두 표시하면 핵심 강점과 부족 역량을 빠르게 찾기 어려웠다.

#### 해결

역량 분석에는 키워드, 상태, 짧은 신호, 다음 행동만 남겼다. 상세 경험 연결과 예상 질문은 준비 항목으로 이동해 정보의 깊이를 유지하면서 화면의 역할을 분리했다.

### 28.5 E2E 테스트의 동일 문구 중복 문제

#### 문제

상태 이름이 요약 범례와 상태별 영역에 동시에 표시되어 Playwright의 strict mode에서 동일 텍스트가 여러 개 검색되었다.

#### 해결

역량 분석 섹션 내부에서 상태 영역을 기준으로 locator 범위를 제한해 실제 카드 목록을 검증하도록 수정했다.

### 28.6 실제 GMS 호출에서 응답 본문이 비어 있던 문제

#### 문제

`gpt-5-nano`에 `max_completion_tokens=8000`을 지정했지만, 기본 추론 설정에서 8,000 토큰을 모두 reasoning token으로 사용했다.

```text
finish_reason: length
completion_tokens: 8000
reasoning_tokens: 8000
content length: 0
```

HTTP 응답 자체는 성공했지만 실제 JSON 본문이 없어 빈 분석 결과로 정규화되었다.

#### 해결

GMS 요청에 다음 옵션을 명시했다.

```json
{
  "max_completion_tokens": 8000,
  "reasoning_effort": "minimal"
}
```

이를 통해 추론 토큰 소진을 방지하고 구조화된 JSON 본문을 안정적으로 생성하도록 했다. 환경에 따라 조정할 수 있도록 다음 환경변수도 지원한다.

- `GMS_MAX_COMPLETION_TOKENS`
- `GMS_REASONING_EFFORT`

### 28.7 실제 호출 검증 스크립트에서 한글 입력이 손상된 문제

#### 문제

PowerShell here-string을 Python 표준 입력으로 전달하는 과정에서 한글 딕셔너리 키가 `?`로 변환되었다. 이 때문에 실제 GMS 프롬프트에는 `전공`, `프로젝트`, `자소서` 등이 `미입력`으로 표시되었다.

#### 해결

실제 서비스와 동일한 한글 키를 UTF-8 JSON 파일로 생성한 후 Python에서 `encoding="utf-8"`로 읽어 요청했다. GMS 호출 전에 프로젝트명, 성과, 기업명, 역기구학, 모션 플래닝, EtherCAT이 최종 프롬프트에 포함되는지도 검사했다.

최종 호출에서는 다음 결과가 확인되었다.

- `역기구학 및 제어 알고리즘`: 강점
- `모션 플래닝 및 궤적 생성`: 답변 정리
- `ROS2 기반 로봇 소프트웨어`: 강점
- `실시간 제어 및 EtherCAT`: 정보 또는 경험 부족
- `산업용 로봇 제어 소프트웨어`: 답변 정리

## 29. 최종 사용자 경험

개선된 결과 화면에서 사용자는 먼저 역량 분석을 통해 전체 준비 상태를 빠르게 확인한다.

```text
어필 가능
- 로봇 제어
- 문제 해결

답변 정리
- 모션 플래닝

학습 필요
- EtherCAT
- 산업용 네트워크
```

이후 준비 항목에서 각 키워드의 구체적인 준비 방법을 확인한다.

```text
로보틱스 → 역기구학
- 내 로봇팔 프로젝트를 어필
- 제어 방식과 성능 개선 수치를 정리
- 예상 질문으로 답변 연습

로보틱스 → 모션 플래닝
- 경로 생성 경험을 직무 용어로 재구성
- 최적화 기준과 충돌 회피 판단을 정리

산업용 통신 → EtherCAT
- 직접 경험 없음
- 동작 원리와 실시간 제어에서의 장점을 학습
- 유사 통신 경험과 빠른 학습 과정을 연결해 답변
```

이 구조를 통해 역량 분석은 “현재 상태를 판단하는 화면”, 준비 항목은 “실제로 공부하고 답변을 만드는 화면”으로 기능한다.

## 30. 준비 항목 2차 구조 개선

### 30.1 개선 목적

기존 준비 항목은 `로보틱스`, `산업용 통신` 같은 지식 분야를 최상위 분류로 사용했다. 이 구조에서는 채용공고의 담당업무가 모두 반영되었는지 확인하기 어렵고, 어떤 업무를 먼저 준비해야 하는지도 명확하지 않았다.

개선 구조는 채용공고의 담당업무를 출발점으로 사용한다.

```text
담당업무 핵심 키워드
├─ 경험·역량 기반 우선순위
├─ 채용공고 담당업무 원문
├─ 연결 경험·활용 역량
└─ 직무 지식 키워드
   ├─ 업무 연결
   ├─ 내 연결점: 근거·전환·보완
   ├─ 핵심 개념과 확인 기준
   ├─ 준비 순서
   └─ 예상 질문: 개념·경험·적용
```

### 30.2 담당업무 전체 커버리지

채용공고의 `담당업무`, `주요업무`, `수행업무`, `업무내용` 영역을 서버에서 먼저 추출한다.

```text
1. 산업용 로봇 제어 알고리즘 개발
2. 로봇 매니퓰레이터의 역기구학 및 궤적 생성
3. 충돌 회피를 포함한 모션 플래닝
4. EtherCAT 기반 서보 모터 실시간 제어
```

추출된 업무마다 `responsibility_index`를 부여하고 GMS가 각 업무를 하나 이상의 최상위 category로 생성하도록 했다. 업무 개수에는 고정 상한을 두지 않는다.

GMS가 업무를 합치거나 누락하면 서버가 다음 품질 조건을 검사한다.

- 추출된 모든 담당업무가 결과에 존재하는가
- 각 담당업무에 직무 지식 subtopic이 존재하는가
- 각 subtopic에 예상 질문이 3개 이상 존재하는가

조건을 충족하지 못하면 누락된 담당업무만 별도로 GMS에 요청해 기존 결과에 병합한다.

### 30.3 경험·역량 기반 우선순위

각 담당업무는 다음 정보를 이용해 정렬한다.

1. 직접 연결되는 프로젝트·경력·자기소개서 경험
2. 업무에 전환할 수 있는 유사 경험
3. 보유 기술과 역량 키워드
4. 필수 업무 여부
5. 경험이 없을 때 학습 시급성

출력 필드는 다음과 같다.

```json
{
  "category": "모션 플래닝",
  "responsibility_index": 3,
  "responsibility": "충돌 회피를 포함한 모션 플래닝",
  "priority": 2,
  "priority_reason": "A* 경로 탐색 경험을 전환할 수 있지만 구성공간과 충돌 검사 보완이 필요함",
  "experience_match": "related",
  "experience_keywords": ["A* 경로 탐색", "장애물 회피"],
  "competency_keywords": ["경로 계획", "비용 함수", "ROS2"]
}
```

`experience_match`는 다음 세 상태만 사용한다.

- `direct`: 직접 업무 경험
- `related`: 유사하거나 전환 가능한 경험
- `none`: 연결 경험 없음

GMS가 채용공고 키워드를 지원자 경험으로 잘못 분류하지 않도록, 서버는 `experience_keywords`를 실제 프로필·자기소개서 내용과 다시 대조한다. 입력에서 확인되지 않는 키워드는 제거한다.

### 30.4 직무 지식별 체계적 연결 분석

`왜 준비해야 하는가`는 일반적인 중요성 설명이 아니라 해당 지식이 담당업무의 어떤 구현·판단·검증에 사용되는지 작성한다.

`내 연결점`은 다음 세 요소로 분리한다.

| 요소 | 내용 |
|---|---|
| 근거 | 프로필·자기소개서에서 확인한 행동, 기술, 결과 |
| 전환 | 해당 경험을 현재 담당업무에 활용할 수 있는 이유 |
| 보완 | 직접 업무와 비교했을 때 추가 학습하거나 설명할 부분 |

예시:

```json
{
  "experience_connection": {
    "evidence": "ROS2 환경에서 A* 경로 탐색과 장애물 회피를 구현함",
    "transferable_point": "상태 공간 탐색과 비용 함수 설계 경험을 모션 플래닝에 연결 가능",
    "gap": "매니퓰레이터 구성공간과 RRT 계열 알고리즘 경험은 없음"
  }
}
```

### 30.5 핵심 개념 강화

기존의 단순 문자열 목록을 `keyword`와 `checkpoint` 구조로 변경했다.

```json
{
  "study_focus": [
    {
      "keyword": "Configuration Space",
      "checkpoint": "작업공간과의 차이와 자유도 증가 영향을 설명"
    },
    {
      "keyword": "A*와 RRT",
      "checkpoint": "완전성, 최적성, 계산 비용을 비교"
    }
  ]
}
```

화면에서는 개념명과 함께 면접 준비 완료 기준을 표시한다. 단순히 무엇을 읽을지가 아니라 어느 수준까지 설명할 수 있어야 하는지 확인할 수 있다.

### 30.6 준비 순서

`approach` 한 문장을 `preparation_steps` 배열로 변경했다.

- 어필: 경험 흐름 정리 → 기술 선택 이유 → 성과와 검증 연결
- 답변 정리: 부족 개념 보완 → 유사 경험 전환 → 직접 경험과 차이 설명
- 학습: 기초 원리 → 비교 기준 → 담당업무 적용 → 질문 답변 연습

### 30.7 예상 질문 확대

각 직무 지식마다 최소 세 종류의 질문을 생성한다.

| 유형 | 목적 |
|---|---|
| `concept` | 개념과 동작 원리 확인 |
| `experience` | 지원자의 경험과 직무 지식 연결 확인 |
| `application` | 담당업무 적용, 선택 기준, 트레이드오프 확인 |

질문 총개수에 고정 상한을 두지 않고, 모든 담당업무에 대해 최소 하나 이상의 직무 지식과 질문 묶음을 생성한다.

화면에서는 질문 문장을 먼저 빠르게 확인하고, `답변 방향`을 펼쳤을 때 답변 순서와 꼬리질문을 확인하도록 변경해 많은 질문이 생성되어도 가독성을 유지한다.
