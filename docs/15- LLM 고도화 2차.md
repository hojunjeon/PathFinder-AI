# PathFinder AI: LLM 고도화 2차

## 1. 개선 목적

이번 고도화는 `14-2_LLM파트_분석.md`의 설계를 바탕으로, 분석 결과가 단순한 직무 키워드 추천이 아니라 다음 질문에 답하도록 개선하는 데 목적이 있다.

1. 이 기업의 이 직무에 지원할 때 나의 어떤 경험이 강점인가?
2. 경험은 있지만 자기소개서나 면접 답변으로 정리가 부족한 역량은 무엇인가?
3. 채용공고가 요구하지만 현재 지식·경험 근거가 부족한 영역은 무엇인가?
4. 부족한 영역은 어떤 개념부터 공부해야 하는가?
5. 내 프로젝트·경력·자기소개서 내용을 직무 업무와 어떻게 연결해 답변해야 하는가?

점수나 합격 가능성을 보여주는 대신, 채용공고와 사용자 입력에서 확인한 근거를 중심으로 **강점 어필 → 경험 정리 → 지식 학습 → 답변 연습**의 흐름을 제공하도록 변경했다.

---

## 2. 주요 변경 사항

### 2.1 단일 프롬프트를 2단계 LLM 파이프라인으로 분리

기존에는 한 번의 요청에서 역량 진단과 상세 준비 항목을 모두 생성했다. 이 방식은 입력과 출력이 길어질수록 역량 분류와 경험 연결의 일관성이 떨어질 수 있었다.

변경 후에는 다음 두 단계로 분리했다.

#### 1단계: 직무 역량 진단

- 채용공고의 담당업무, 필수 요건, 우대 요건 추출
- 사용자 프로필, 구조화된 이력서, 프로젝트, 자기소개서와 대조
- 핵심 역량을 `강점`, `정리할 역량`, `약점`으로 분류
- 각 판단에 사용자 경험 또는 채용공고 근거 저장
- 입력이 부족한 항목은 억지로 약점 처리하지 않고 별도의 경고로 분리

#### 2단계: 준비 항목 생성

- 1단계 진단 결과를 다시 입력으로 사용
- 채용공고의 각 담당업무를 빠짐없이 준비 항목으로 생성
- 담당업무별 필요한 직무 지식을 세분화
- 각 지식에 사용자 경험을 직접 경험, 유사 경험, 경험 없음으로 연결
- 공부할 개념, 준비 순서, 예상 질문, 답변 관점을 생성

이 구조를 통해 2단계 결과가 1단계 역량 판단과 일치하도록 만들었다.

---

### 2.2 면접 유형 의존 제거

기존 프롬프트에는 사용자가 선택한 기술 면접, PT 면접, 인성 면접 등의 유형 정보가 포함되어 있었다.

고도화된 LLM 분석에서는 면접 형식보다 다음 정보를 우선한다.

- 채용공고 원문과 담당업무
- 필수 역량과 우대사항
- 사용자의 프로젝트·경력·자격·수상 경험
- 자기소개서에 작성된 역할, 행동, 기술, 결과
- 기업과 직무의 업무 맥락

선택한 면접 유형은 기존 서비스 흐름과 화면 표시를 위해 유지하지만, LLM의 직무 역량 판단과 질문 생성에는 사용하지 않는다.

---

### 2.3 역량 분류를 3가지 사용자 행동으로 통일

#### 강점

- 채용공고의 업무와 직접 연결되는 경험·지식 근거가 있는 역량
- 결과 화면에서 경험, 입력 근거, 공고 연결 이유, 답변 초점을 제공
- 사용자는 무엇을 자신 있게 말해야 하는지 확인할 수 있다.

#### 정리할 역량

- 관련 경험은 있지만 역할, 선택 이유, 성과, 직무 용어 연결이 부족한 역량
- 단순 약점이 아니라 서비스의 핵심 차별화 영역으로 분리
- 연결 경험, 빠진 설명, 답변 정리 방법을 제공
- 사용자는 이미 한 경험을 직무 역량으로 재구성할 수 있다.

#### 약점

- 채용공고가 요구하지만 사용자 입력에서 관련 지식 또는 경험 근거를 확인하기 어려운 역량
- 부족한 이유, 공고 근거, 다음 학습 행동을 제공
- 사용자는 무엇부터 공부하거나 경험을 보완해야 하는지 확인할 수 있다.

기존 `articulate`, `study` 데이터도 각각 `organize`, `weakness`로 변환해 과거 분석 결과와의 호환성을 유지했다.

---

### 2.4 준비 항목 구조 고도화

준비 항목을 다음 구조로 변경했다.

```text
채용공고 담당업무
└─ 직무 핵심 역량
   └─ 세부 직무 지식
      ├─ 업무 연결
      ├─ 내 경험 근거
      ├─ 직무로 전환 가능한 이유
      ├─ 직접 업무와 비교한 보완점
      ├─ 핵심 학습 개념과 확인 기준
      ├─ 실제 준비 순서
      ├─ 답변 관점
      └─ 개념·경험·적용 예상 질문
```

세부 지식마다 다음 데이터가 생성된다.

| 항목 | 설명 |
|---|---|
| `task_connection` | 해당 지식이 실제 담당업무의 판단·구현·검증에 쓰이는 방식 |
| `experience_match` | 프로필 또는 자기소개서에서 연결되는 경험 |
| `experience_connection.evidence` | 입력에서 확인한 행동·기술·결과 |
| `experience_connection.transferable_point` | 기존 경험을 지원 직무에 전환해 설명할 수 있는 이유 |
| `experience_connection.gap` | 직접 업무와 비교했을 때 추가로 보완할 부분 |
| `core_concepts` | 우선 공부할 핵심 개념 |
| `checkpoint` | 면접에서 설명·비교·적용할 수 있어야 하는 학습 기준 |
| `preparation_steps` | 경험 정리 또는 학습의 실제 수행 순서 |
| `appeal_perspective` | 내 경험을 기업·직무 가치로 연결하는 답변 관점 |
| `questions` | 개념, 경험, 적용 관점의 개인화 예상 질문 |

## 3. 분석 결과 페이지 변화

### 3.1 직무 역량 진단 영역

기존의 상태 카드 중심 화면을 다음과 같이 확장했다.

- 상단에서 `강점 / 정리할 역량 / 약점` 개수와 분포 표시
- 각 역량의 짧은 판단 근거와 다음 행동 표시
- 강점 상세 근거 표시
  - 내 경험
  - 프로필·자기소개서 근거
  - 채용공고 연결
  - 답변 초점
- 정리할 역량 상세 표시
  - 연결 가능한 경험
  - 현재 답변에서 빠진 설명
  - 직무 언어로 정리하는 방법
- 약점 상세 표시
  - 지식 부족 또는 경험 부족 구분
  - 채용공고 요구 근거
  - 우선 학습·연습 행동

### 3.2 담당업무별 준비 항목

기존의 업무 연결, 경험 연결, 핵심 개념, 준비 순서, 예상 질문에 더해 **답변 관점**을 추가했다.

답변 관점은 사용자가 단순히 기술 개념을 설명하는 데 그치지 않고 다음 흐름으로 답하도록 돕는다.

```text
내가 해결한 문제
→ 사용한 기술과 선택 이유
→ 내가 담당한 행동
→ 검증한 결과
→ 지원 직무의 업무에 기여할 수 있는 방식
```

## 4. 데이터 구조 변화

### 4.1 1단계 진단 응답

```json
{
  "competency_analysis": {
    "summary": "요약",
    "competency_map": [
      {
        "keyword": "핵심 역량",
        "status": "strength | organize | weakness",
        "importance": "required | preferred",
        "signal": "판단 요약",
        "action": "다음 행동"
      }
    ],
    "strength_details": [],
    "organize_details": [],
    "weakness_details": [],
    "required_competencies": [],
    "input_warnings": []
  }
}
```

### 4.2 2단계 준비 항목 응답

```json
{
  "text_roadmap": "전체 준비 순서 요약",
  "preparation_roadmap": [
    {
      "category": "담당업무 핵심 역량",
      "responsibility": "채용공고 담당업무 원문",
      "experience_match": "direct | related | none",
      "sub_knowledges": [
        {
          "knowledge_keyword": "세부 직무 지식",
          "preparation_type": "appeal | organize | study",
          "task_connection": "업무 연결",
          "experience_connection": {},
          "core_concepts": [],
          "preparation_steps": [],
          "appeal_perspective": "답변 관점",
          "questions": []
        }
      ]
    }
  ]
}
```

백엔드 저장 구조는 기존 `competency_gap`, `timeline_data`를 유지한다. LLM 서버에서 새 응답을 기존 API 계약으로 정규화하기 때문에 데이터베이스 마이그레이션 없이 적용할 수 있다.

## 7. 사용자 관점의 최종 변화

고도화 전에는 사용자가 직무 역량 키워드와 예상 질문을 확인하는 데 초점이 있었다.

고도화 후에는 다음 순서로 지원 전략을 이해할 수 있다.

1. **강점 발견**: 내 프로젝트·경력 중 채용공고 업무와 직접 연결되는 경험을 확인한다.
2. **경험 재정리**: 해본 경험이지만 설명이 부족한 부분을 역할·선택 이유·성과·직무 기여 관점으로 정리한다.
3. **약점 학습**: 공고가 요구하지만 근거가 없는 지식과 경험을 구분하고 우선 학습한다.
4. **업무별 연결**: 채용공고의 각 담당업무에 내 경험이 직접, 유사, 미연결 중 어디에 해당하는지 확인한다.
5. **실전 답변 준비**: 핵심 개념과 사용자 경험을 함께 사용하는 개인화 질문과 답변 관점으로 연습한다.

결과적으로 사용자는 단순히 “무엇을 공부할지”뿐 아니라, **왜 공부해야 하는지, 내 어떤 경험을 연결할지, 면접에서 어떤 논리로 답할지**까지 한 화면에서 확인할 수 있다.
