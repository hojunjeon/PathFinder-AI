# 로그인·회원가입 계정 생성 기능 개선

작성일: 2026-06-24

## 1. 작업 목적

기존 로그인 페이지의 탭 전환 구조를 유지하면서 회원가입에 필요한 최소 계정 정보를 입력받고, 실제 사용자 계정과 초기 프로필이 생성되도록 인증 기능을 개선했다.

개인 이력, 전공, 학력, 경력 등의 상세 정보는 기존 프로필 페이지에서 관리하므로 회원가입 화면에는 포함하지 않았다.

## 2. 구현된 기능

### 2.1 로그인과 회원가입의 단일 페이지 유지

- 기존 `/login` 페이지를 그대로 사용한다.
- 사용자는 페이지 이동 없이 `로그인`과 `회원가입` 탭을 전환할 수 있다.
- 로그인 방식과 로그인 성공 후 메인 페이지로 이동하는 기존 동작은 유지했다.

### 2.2 회원가입 입력 항목 추가

회원가입 탭에서 다음 항목을 입력받는다.

| 항목 | 용도 | 검증 |
|---|---|---|
| 이름 | 계정 생성 시 초기 프로필 이름으로 저장 | 필수, 최대 50자 |
| 이메일 | 로그인 계정 식별자 | 필수, 이메일 형식, 중복 불가 |
| 비밀번호 | 계정 인증 정보 | 필수, 최소 8자 |
| 비밀번호 확인 | 비밀번호 입력 오류 방지 | 필수, 비밀번호와 동일해야 함 |

로그인 탭에서는 기존과 동일하게 이메일과 비밀번호만 표시한다.

### 2.3 비밀번호 확인 검증

비밀번호 일치 여부를 프론트엔드와 백엔드에서 모두 확인한다.

- 프론트엔드: 값이 다르면 API를 호출하지 않고 오류 메시지를 표시한다.
- 백엔드: 클라이언트 검증을 우회한 요청도 거부한다.
- 오류 메시지: `비밀번호가 일치하지 않습니다.`

프론트엔드 검증은 불필요한 네트워크 요청을 줄이고, 백엔드 검증은 데이터 무결성을 보장한다.

### 2.4 실제 회원가입 처리

회원가입 요청이 성공하면 다음 작업이 수행된다.

1. Django 사용자 계정을 생성한다.
2. 비밀번호를 평문이 아닌 Django 비밀번호 해시 방식으로 저장한다.
3. 사용자와 연결된 `Profile` 레코드를 생성한다.
4. 회원가입 화면에서 입력한 이름을 `Profile.name`에 초기값으로 저장한다.
5. JWT access/refresh 토큰을 발급한다.
6. 프론트엔드가 토큰을 `localStorage`에 저장하고 로그인 상태로 전환한다.
7. 메인 페이지(`/`)로 이동한다.

## 3. 프론트엔드 구현

### 3.1 회원가입 폼

수정 파일:

- `frontend/src/views/LoginView.vue`

회원가입 모드에서만 이름과 비밀번호 확인 입력창을 조건부 렌더링하도록 구현했다.

```vue
<div v-if="mode === 'signup'" class="field">
  <label for="name">이름</label>
  <input id="name" v-model.trim="name" type="text" required />
</div>

<div v-if="mode === 'signup'" class="field">
  <label for="password-confirm">비밀번호 확인</label>
  <input
    id="password-confirm"
    v-model="passwordConfirm"
    type="password"
    required
  />
</div>
```

비밀번호 입력창의 자동 완성 속성도 현재 모드에 맞게 구분했다.

- 로그인: `current-password`
- 회원가입: `new-password`

### 3.2 제출 전 검증

회원가입 모드에서 비밀번호와 비밀번호 확인 값이 다르면 요청을 중단한다.

```javascript
if (mode.value === 'signup' && password.value !== passwordConfirm.value) {
  errorMsg.value = '비밀번호가 일치하지 않습니다.'
  return
}
```

### 3.3 회원가입 API 요청

수정 파일:

- `frontend/src/stores/auth.js`

여러 개의 위치 인자 대신 하나의 객체로 회원가입 정보를 전달하도록 구성했다.

```javascript
async function signup({ name, email, password, passwordConfirm }) {
  const { data } = await api.post('/api/auth/signup/', {
    name,
    email,
    password,
    password_confirm: passwordConfirm,
  })
}
```

실제 요청 데이터는 다음과 같다.

```json
{
  "name": "홍길동",
  "email": "new@example.com",
  "password": "password123!",
  "password_confirm": "password123!"
}
```

## 4. 백엔드 구현

수정 파일:

- `backend/accounts/serializers.py`

기존 `SignupSerializer`에 이름과 비밀번호 확인 필드를 추가했다.

```python
class SignupSerializer(serializers.ModelSerializer):
    name = serializers.CharField(write_only=True, max_length=50)
    password = serializers.CharField(write_only=True, min_length=8)
    password_confirm = serializers.CharField(write_only=True)
```

### 4.1 비밀번호 일치 검증

```python
def validate(self, data):
    if data['password'] != data['password_confirm']:
        raise serializers.ValidationError({
            'password_confirm': '비밀번호가 일치하지 않습니다.'
        })
    return data
```

### 4.2 사용자와 초기 프로필 생성

`name`과 `password_confirm`은 `User` 모델의 필드가 아니므로 사용자 생성 전에 분리한다.

```python
def create(self, validated_data):
    name = validated_data.pop('name').strip()
    validated_data.pop('password_confirm')
    user = User.objects.create_user(**validated_data)
    Profile.objects.create(user=user, name=name)
    return user
```

이름은 기존 `Profile.name` 필드를 사용하므로 새 모델 필드나 데이터베이스 마이그레이션은 추가하지 않았다.

## 5. API 동작

### 요청

```http
POST /api/auth/signup/
Content-Type: application/json
```

### 성공 응답

- HTTP 상태: `201 Created`
- 응답 데이터: JWT access 토큰과 refresh 토큰

```json
{
  "access": "<access-token>",
  "refresh": "<refresh-token>"
}
```

### 실패 조건

- 필수 입력 누락
- 이메일 형식 오류
- 이미 가입된 이메일
- 8자 미만 비밀번호
- 비밀번호와 비밀번호 확인 불일치

## 8. 설계 원칙

### DRY

- 사용자 생성과 비밀번호 해시는 기존 `UserManager.create_user()`를 재사용했다.
- 로그인과 회원가입 성공 후 토큰 저장 로직은 기존 인증 스토어 흐름을 유지했다.
- 이름 저장을 위해 별도 계정 필드를 만들지 않고 기존 `Profile.name`을 사용했다.

### KISS

- 새로운 회원가입 페이지나 별도 인증 컴포넌트를 만들지 않았다.
- 기존 탭과 폼에 필요한 필드만 조건부로 추가했다.
- 기존 회원가입 API를 확장했으며 새로운 엔드포인트를 추가하지 않았다.

### YAGNI

- 주소, 전화번호, 전공, 학력, 경력 등의 상세 개인정보는 추가하지 않았다.
- 이메일 인증, 소셜 회원가입, 비밀번호 복구 등 요청되지 않은 기능은 구현하지 않았다.
- 기존 모델로 요구사항을 충족할 수 있어 DB 마이그레이션을 만들지 않았다.

## 9. 변경 범위
