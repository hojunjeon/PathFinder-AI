# 홈 내비게이션 및 자기소개서 확인 UI

작성일: 2026-06-24  
작업 브랜치: `fix/everthing`

> 최종 상태 안내: 이 문서의 3~9절은 초기 구현과 개선 과정을 기록한 내용이다. 현재 적용된 최종 설계·데이터 구조·트러블슈팅 결과는 **10절**을 기준으로 한다.

## 1. 작업 개요

이번 작업에서는 사용자가 주요 화면으로 더 쉽게 이동할 수 있도록 다음 두 가지 내비게이션 기능을 추가했다.

1. 로그인 후 표시되는 상단 메뉴에 `홈` 버튼 추가
2. 분석 결과 페이지의 왼쪽 사이드바에 `자기소개서 입력 화면` 버튼 추가

분석 결과를 생성하거나 계산하는 로직은 변경하지 않았다. 기존 Vue Router 경로와 분석 생성 화면을 재사용하여 화면 이동 기능만 구현했다.

## 2. 상단 메뉴 홈 버튼

### 구현 목적

기존 상단 메뉴에는 다음과 같은 기능 메뉴가 있었지만, 메인 화면으로 돌아가는 명시적인 `홈` 메뉴가 없었다.

- 로드맵 생성
- 히스토리
- 면접 후기
- 프로필
- 채용시장 분석

서비스 로고를 클릭해도 홈으로 이동할 수 있지만, 사용자가 기능을 바로 이해할 수 있도록 상단 메뉴에 별도의 `홈` 버튼을 추가했다.

### 수정 파일

- `frontend/src/App.vue`

### 구현 방식

로그인 사용자에게 표시되는 전역 내비게이션에 다음 Vue Router 링크를 추가했다.

```vue
<router-link to="/">홈</router-link>
```

`router-link`를 사용했기 때문에 전체 페이지를 새로 불러오지 않고 Vue Router가 `/` 경로의 `HomeView.vue`로 이동한다.

기존 전역 내비게이션 CSS를 그대로 사용하므로 별도의 색상이나 레이아웃 스타일은 추가하지 않았다. 기존 메뉴와 동일하게 다음 동작도 적용된다.

- 마우스를 올렸을 때 메뉴 글자 강조
- 현재 홈 경로에 있을 때 활성 메뉴 스타일 적용
- 모바일 화면에서 기존 메뉴와 함께 줄바꿈

## 3. 분석 결과 페이지의 자기소개서 입력 화면 버튼

### 구현 목적

분석 결과를 확인한 사용자가 자기소개서를 다시 입력하거나 새 분석을 시작하려면 기존에는 상단의 `로드맵 생성` 메뉴를 사용해야 했다.

결과 페이지 안에서도 입력 화면으로 바로 이동할 수 있도록 왼쪽 사이드바의 `섹션 이동` 영역 아래에 `자기소개서 입력 화면` 버튼을 추가했다.

### 수정 파일

- `frontend/src/views/AnalyzeResultView.vue`

### 구현 방식

분석 결과 페이지의 사이드바에 다음 링크를 추가했다.

```vue
<router-link class="cover-letter-link" to="/analyze/new">
  자기소개서 입력 화면
</router-link>
```

버튼을 누르면 기존 분석 생성 화면인 `/analyze/new`로 이동한다.

분석 생성 화면은 다음 순서로 구성되어 있으므로, 사용자는 채용공고 정보를 입력한 후 기존 자기소개서 입력 단계로 이동할 수 있다.

1. 채용공고 입력
2. 자기소개서 입력

이번 변경에서는 특정 분석 결과에 저장된 자기소개서 내용을 결과 화면에 표시하거나 자동으로 입력하지 않았다. 요청 범위에 맞게 기존 자기소개서 입력 화면으로 이동하는 기능만 구현했다.

### 버튼 스타일

기존 디자인 토큰과 사이드바 스타일을 재사용하여 다음과 같이 구성했다.

- 최소 높이 `44px`로 클릭 영역 확보
- 기존 `border`, `radius`, `surface` 색상 변수 사용
- 사이드바 너비에 맞춘 중앙 정렬
- hover 시 테두리와 글자를 accent 색상으로 강조
- 다크 모드에서도 기존 CSS 변수에 따라 자동 대응

새로운 색상이나 별도의 디자인 시스템은 추가하지 않았다.

## 9. 자기소개서 확인 UX 재검토 및 변경

### 문제 정의

기존 `자기소개서 입력 화면` 버튼은 결과 페이지에서 `/analyze/new`로 이동했다. 그러나 사용자가 결과를 보다가 원하는 행동은 새 자기소개서를 작성하거나 수정하는 것이 아니라, **현재 분석에 어떤 자기소개서가 사용되었는지 확인하는 것**에 가깝다.

기존 방식에는 다음 문제가 있었다.

- 현재 분석 결과 화면을 벗어나 사용 맥락이 끊긴다.
- 새 분석의 첫 단계인 채용공고 입력 화면으로 이동해 확인 목적과 입력 목적이 섞인다.
- 분석 당시 제출본이 자동으로 채워지지 않아 실제로는 내용을 바로 확인할 수 없다.
- 수정 가능성이 낮은 읽기 전용 정보에 별도 페이지 이동을 사용해 동작이 과하다.

### UI 대안 분석

| 대안 | 장점 | 단점 | 판단 |
|---|---|---|---|
| 별도 입력 페이지 이동 | 기존 화면 재사용 | 결과 맥락 이탈, 제출본 즉시 확인 불가, 수정 기능처럼 보임 | 제외 |
| 결과 본문에 항상 노출 | 접근이 빠름 | 최대 12,000자의 긴 원문이 핵심 분석 내용을 밀어냄 | 제외 |
| 접기/펼치기 영역 | 페이지 이탈 없음 | 긴 원문을 펼치면 결과 페이지의 탐색 길이가 크게 증가 | 보류 |
| 읽기 전용 모달 | 현재 맥락 유지, 필요할 때만 노출, 수정 기능과 분리 | 긴 내용에 내부 스크롤 필요 | 채택 |

### 최종 결정

분석 결과 사이드바의 이동 링크를 제거하고 `제출 자기소개서 확인` 버튼으로 교체했다. 버튼을 누르면 현재 분석을 생성할 때 제출한 자기소개서를 읽기 전용 모달로 표시한다.

이 방식은 다음 사용자 흐름을 만든다.

```text
분석 결과 확인
→ 제출 자기소개서 확인
→ 분석 당시 제출본을 모달에서 읽음
→ 닫기
→ 같은 분석 위치에서 계속 확인
```

수정 기능과 새 분석 생성 기능은 의도적으로 연결하지 않았다. 사용자가 실제로 새 분석을 만들고 싶을 때는 전역 내비게이션의 `로드맵 생성`을 사용한다.

### 모달 UI 구성

- 제목: `제출 자기소개서`
- 안내: `이 분석을 생성할 때 제출한 내용을 읽기 전용으로 보여줍니다.`
- 본문: 줄바꿈을 보존한 자기소개서 원문
- 상태 문구: `분석 당시 제출본 · 읽기 전용`
- 닫기 방식:
  - 우측 상단 닫기 버튼
  - 하단 `닫기` 버튼
  - Escape 키
  - 모달 바깥 영역 클릭
- 긴 내용은 모달 본문만 독립 스크롤
- 데스크톱과 모바일 모두 화면 중앙에 모달 표시
- `Q.`와 `A.` 형식의 원문을 항목별 카드로 나누고, 질문과 답변 영역을 시각적으로 구분
- 기존 형식과 다른 원문은 내용 유실 없이 `제출 내용` 단일 카드로 표시
- 제출 자기소개서가 없으면 버튼 대신 `제출된 자기소개서가 없습니다.` 표시

### 접근성

- 네이티브 `<dialog>`를 사용해 모달이 열렸을 때 포커스를 내부에 제한한다.
- `aria-labelledby`와 `aria-describedby`로 제목·설명을 연결한다.
- 본문 스크롤 영역은 키보드 포커스를 받을 수 있다.
- 모달을 닫으면 `제출 자기소개서 확인` 버튼으로 포커스를 돌려준다.

### API 및 개인정보 범위

자기소개서 원문은 기존에도 `Analysis.submitted_cover_letter`에 분석별로 저장되고 있었지만, 결과 조회 serializer에는 포함되지 않았다.

다음과 같이 조회 계약을 분리했다.

- 분석 상세 `/api/analyze/:id/`: `submitted_cover_letter` 포함
- 분석 생성 응답: 기존과 동일하게 원문 미포함
- 분석 히스토리 `/api/analyze/history/`: 원문 미포함

상세 API는 기존과 같이 `Analysis.objects.get(pk=analysis_id, user=request.user)` 조건을 사용하므로 본인 분석만 조회할 수 있다. 히스토리 목록에 원문을 싣지 않아 개인정보 노출 범위와 목록 응답 크기도 제한했다.

사용자 피드백을 반영해 모달 위치와 본문 표현을 추가 개선했다.

- 모바일을 포함한 모든 화면에서 모달을 화면 중앙에 배치
- 저장된 `Q.`/`A.` 구분을 파싱해 자기소개서 항목별 카드 생성
- 각 카드에 `자기소개서 N`, `항목`, `답변` 레이블을 두고 질문과 답변 본문을 별도 영역으로 표시
- 제목 아래에서 바로 자기소개서 카드가 시작되도록 안내 문구와 하단 상태 영역 제거
- 여러 항목의 분리 표시와 모달 중심 좌표를 Playwright에서 검증

### 저장 원문 그대로 표시

자기소개서 항목과 답변은 DB의 `submitted_cover_letter`에 하나의 원문으로 정상 저장된다. 모달에서는 이 문자열을 항목별로 다시 해석하거나 정제하지 않는다.

- 상세 API의 `submitted_cover_letter`를 그대로 출력한다.
- 사용자가 입력한 `Q.`, `A.`, 문장, 빈 줄과 문단 구성을 유지한다.
- `<pre>`와 `white-space: pre-wrap`을 사용해 긴 문장은 화면 폭에 맞춰 줄바꿈하되 저장된 개행은 보존한다.
- 별도 파싱이 없으므로 답변이 항목 분리 과정에서 누락될 가능성이 없다.

### 입력 구조 보존 및 가독성 개선

입력 화면에는 항목과 답변 칸이 분리되어 있었지만 기존 제출 과정에서는 두 값을 `Q. ...\nA. ...` 원문 하나로 합친 뒤 구조화된 배열을 버리고 있었다. 따라서 결과 화면에서 구분 표시하려면 합쳐진 문자열을 다시 해석해야 했다.

이를 다음과 같이 변경했다.

- `submitted_cover_letter`: 기존 원문을 그대로 유지해 LLM 입력과 과거 호환성 보장
- `submitted_cover_letter_items`: 입력 당시 `{ question, answer }` 배열을 별도로 저장
- 모달: 항목은 굵은 제목 영역, 답변은 일반 본문 영역으로 표시
- 답변 내부 문단과 줄바꿈은 그대로 보존
- 모달 제목은 고정하고 본문 영역에 세로 이동바를 제공해 마지막 답변까지 읽을 수 있게 구성
- 기존 분석 데이터는 마이그레이션에서 구조화 배열로 이관
- 구조화할 수 없는 과거 데이터는 원문 표시로 fallback

## 10. 최종 구현 내용 정리

작성일: 2026-06-25

### 10.1 구현한 기능

분석 결과 페이지에서 현재 분석에 사용된 자기소개서를 페이지 이동 없이 다시 확인할 수 있도록 읽기 전용 모달을 구현했다.

최종 사용자 흐름은 다음과 같다.

```text
분석 결과 페이지
→ 사이드바의 제출 자기소개서 확인 버튼 클릭
→ 화면 중앙에 자기소개서 모달 표시
→ 항목과 답변을 구분해서 확인
→ 본문 이동바로 마지막 답변까지 확인
→ 닫기 후 기존 분석 결과를 계속 확인
```

주요 기능:

- 결과 페이지를 벗어나지 않고 모달로 자기소개서 확인
- 자기소개서 항목별 독립 블록 표시
- 항목은 굵은 제목, 답변은 일반 본문으로 시각적 구분
- 답변의 문장, 빈 줄과 문단 구성 보존
- 긴 자기소개서의 세로 스크롤 지원
- 모달 제목은 고정하고 본문만 스크롤
- 데스크톱과 모바일 모두 화면 중앙 배치
- 자기소개서가 없는 분석에는 확인 버튼 대신 빈 상태 안내 표시
- 수정·저장·새 분석 이동 기능은 포함하지 않음

불필요한 상세 안내, 상태 문구와 하단 버튼은 제거하고 다음 요소만 남겼다.

- `제출 자기소개서` 제목
- 닫기 버튼
- 항목별 자기소개서 내용

---

### 10.2 기존 방식의 문제

초기에는 결과 페이지의 `자기소개서 입력 화면` 링크를 누르면 `/analyze/new`로 이동했다.

이 방식은 다음 이유로 사용자 목적과 맞지 않았다.

- 사용자는 수정이 아니라 분석에 사용된 제출본을 확인하려고 한다.
- 새 분석 화면으로 이동하면 현재 분석 결과의 맥락이 끊긴다.
- `/analyze/new`의 첫 화면은 채용공고 입력 단계이므로 자기소개서를 바로 볼 수 없다.
- 분석 당시 제출한 자기소개서가 자동으로 복원되지 않는다.

따라서 입력 화면 이동 링크를 제거하고 현재 분석의 제출본을 보여주는 읽기 전용 모달로 변경했다.

---

### 10.3 자기소개서 데이터 구조

#### 기존 저장 방식

입력 화면은 항목과 답변을 서로 다른 필드로 제공했다.

```text
항목 input
답변 textarea
```

그러나 제출 시점에는 다음 문자열 하나로 합쳐졌다.

```text
Q. 지원동기
A. 사용자가 입력한 답변

Q. 직무 역량
A. 사용자가 입력한 답변
```

이 문자열은 `Analysis.submitted_cover_letter` TextField에 저장됐다.

문장 내용은 보존됐지만 원래 분리되어 있던 `{ question, answer }` 구조는 사라졌다. 결과 화면에서 항목과 답변을 다르게 표현하려면 문자열을 다시 해석해야 했고, 여러 문단과 줄바꿈이 있는 긴 답변을 안정적으로 처리하기 어려웠다.

#### 최종 저장 방식

기존 원문과 구조화 데이터를 함께 저장하도록 변경했다.

```text
submitted_cover_letter
  기존 Q./A. 원문 문자열

submitted_cover_letter_items
  [
    {
      "question": "지원동기",
      "answer": "사용자가 입력한 전체 답변"
    }
  ]
```

두 값을 함께 유지하는 이유:

- `submitted_cover_letter`
  - 기존 LLM 입력 계약 유지
  - 과거 데이터 호환
  - 입력 원문 보존
- `submitted_cover_letter_items`
  - 입력 당시 항목과 답변의 구조 보존
  - 결과 모달에서 별도 문자열 파싱 없이 안정적으로 표시
  - 항목과 답변에 서로 다른 UI 스타일 적용

### 10.8 트러블슈팅

#### 문제 1. 결과 페이지 이동 버튼이 확인 목적과 맞지 않음

증상:

- 자기소개서를 확인하려고 버튼을 눌렀지만 새 분석 화면으로 이동했다.

원인:

- 확인 기능과 새 입력 기능이 동일한 경로를 사용했다.

해결:

- `/analyze/new` 이동 링크 제거
- 결과 페이지 내부 읽기 전용 모달로 교체

---

#### 문제 2. 자기소개서 항목만 보이고 답변이 누락되거나 불안정하게 표시됨

증상:

- 항목은 분리되어 보이지만 긴 답변 또는 여러 문단의 답변이 정상적으로 연결되지 않았다.

원인:

- 입력 당시에는 항목과 답변이 분리되어 있었지만 저장 전에 문자열 하나로 합쳤다.
- 프론트에서 긴 문자열을 정규식으로 다시 분리하는 방식은 줄바꿈 형식과 여러 문단에 의존했다.

해결:

- 입력 당시 `{ question, answer }` 배열을 별도 필드에 그대로 저장
- 프론트 문자열 재파싱 제거
- 상세 API가 반환한 구조화 데이터를 직접 렌더링
- 기존 원문은 LLM 입력과 fallback 용도로 유지

---

#### 문제 3. 원문만 출력하면 항목과 답변의 가독성이 부족함

증상:

- 내용은 모두 보이지만 항목과 답변이 동일한 글씨 스타일로 이어져 빠르게 구분하기 어려웠다.

해결:

- 항목은 굵은 `<h3>` 제목 영역 사용
- 답변은 일반 굵기의 `<p>` 본문 사용
- 항목과 답변 사이에 구분선 적용
- 항목별 카드 사이에 간격 적용
- 답변의 원래 줄바꿈은 `white-space: pre-wrap`으로 유지

---

#### 문제 4. 긴 답변의 아래쪽 내용이 잘림

증상:

- 모달 높이를 넘어가는 답변의 마지막 부분을 볼 수 없었다.
- 초기 CSS에는 `overflow`가 있었지만 실제 스크롤 높이가 생성되지 않았다.

원인:

- 모달 내부 CSS Grid의 자식이 최소 콘텐츠 높이보다 작아지지 않았다.
- 자기소개서 카드가 그리드 행처럼 축소되면서 내용이 잘렸다.
- 카드의 `overflow: hidden`이 잘린 내용을 숨겼다.

해결:

- 모달과 패널에 명시적인 높이와 `overflow: hidden` 적용
- 본문 행에 `minmax(0, 1fr)` 사용
- 패널과 스크롤 영역에 `min-height: 0` 적용
- 카드의 `overflow: hidden` 제거
- 카드 목록을 축소되는 grid 대신 일반 block 흐름으로 변경
- 본문에 `overflow-y: scroll`을 적용해 세로 이동바 항상 제공

회귀 테스트에서는 다음을 직접 측정했다.

```text
scrollHeight > clientHeight
스크롤 후 scrollTop > 0
마지막 자기소개서 카드의 하단이 스크롤 영역 안에 위치
```

---

#### 문제 5. 기존 E2E가 프로필 API 저장을 기대함

증상:

- 전체 E2E 실행 시 자기소개서를 프로필 API에 저장하는 테스트가 실패했다.

원인:

- 현재 제품은 자기소개서를 공통 프로필 데이터가 아니라 분석별 제출 기록으로 저장한다.
- 테스트가 제거된 과거 데이터 계약을 기준으로 작성되어 있었다.

해결:

- 오래된 프로필 저장 E2E 제거
- 분석 생성 요청의 `submitted_cover_letter_items` 저장을 검증
- 분석 상세 모달의 항목, 답변과 이동바 동작을 검증하도록 변경

---

### 10.9 접근성 및 반응형 처리

접근성:

- 네이티브 `<dialog>` 사용
- 모달 제목을 `aria-labelledby`로 연결
- 닫기 아이콘에 `aria-label="자기소개서 닫기"` 적용
- 본문 스크롤 영역에 `tabindex="0"` 적용
- 모달을 닫은 뒤 실행 버튼으로 포커스 복귀
- 항목은 제목 태그를 사용해 문서 구조 제공

반응형:

- 데스크톱: 최대 760px 너비
- 모바일: 화면 좌우 12px 여백
- 데스크톱과 모바일 모두 중앙 정렬
- 모바일에서는 화면 높이에서 32px을 제외한 높이 사용
- 답변 본문은 화면 폭에 맞춰 자동 줄바꿈
