# 분석 결과 면접 준비 표시 정책 및 동작 원리

## 문서 목적

이 문서는 분석 결과 페이지에서 구현하려는 기능, 화면에 보여줄 내용, 그리고 각 기능이 어떤 데이터 흐름과 코드 구조로 동작하는지 나중에 다시 공부할 수 있도록 정리한 기술 문서이다.

중점은 다음 세 가지이다.

1. 분석 결과 페이지 상단과 준비 항목에서 불필요한 설명을 제거한 이유
2. `역량 분석`을 단순 문장 청킹이 아니라 입력 데이터 기반 키워드 요약으로 만들려는 의도
3. 프론트엔드, 백엔드, LLM 서버가 어떤 순서로 데이터를 만들고 화면에 표시하는지

## 구현하고자 한 최종 사용자 경험

사용자가 채용공고와 자기소개서, 프로필 정보를 기반으로 분석을 실행하면 결과 페이지에서 다음 정보를 확인할 수 있어야 한다.

```text
분석 요약
-> 역량 분석
-> 직무 역량 매칭도
-> 준비 항목
```

각 영역의 역할은 다음과 같다.

| 영역 | 역할 |
|---|---|
| 분석 요약 | 회사명, 면접 유형, 현재 준비 진행률을 빠르게 확인 |
| 역량 분석 | 내 프로필·자기소개서·채용공고를 바탕으로 강점, 보완점, 직무 필요 역량을 키워드로 확인 |
| 직무 역량 매칭도 | 강점, 요구 역량, 보완 역량을 점수형 막대로 시각화 |
| 준비 항목 | 실제 면접에서 받을 수 있는 질문, 꼬리질문, 답변 팁을 확인하고 체크 |

핵심 방향은 `설명 많은 학습 계획표`가 아니라 `면접 직전에 바로 볼 수 있는 질문·답변 준비표`이다.

## 주요 변경 목표

### 1. 상단 역량 요약 제거

기존 화면은 가장 상단의 `회사명 면접 준비` 제목 바로 아래에 직무명과 요구 역량 키워드가 함께 표시되었다.

```vue
<div class="hero-keywords">
  <span>{{ analysis.job_title }}</span>
  <span v-for="keyword in heroKeywords" :key="keyword">{{ keyword }}</span>
</div>
```

이 영역은 아래쪽 `역량 분석` 섹션과 정보가 중복되고, 사용자가 처음 화면에 들어왔을 때 핵심 CTA보다 부가 정보가 먼저 보이는 문제가 있었다.

따라서 상단에서는 다음만 남긴다.

- 회사명 기반 제목
- 선택한 면접 유형 chip
- 준비 항목 개수 chip
- 진행률 요약

즉, 역량 키워드는 별도 섹션인 `역량 분석`에서만 보여준다.

### 2. 역량 분석을 입력 근거 기반 키워드로 표시

기존에는 LLM 응답 문자열을 프론트에서 괄호, 콜론, 하이픈, 공백 기준으로 나누어 `키워드 + 설명` 형태처럼 보이게 만들었다.

예를 들어 다음 문자열이 오면,

```text
로봇 제어 SW 개발 역량 - 프로젝트 경험과 채용공고 요구가 연결됩니다
```

프론트에서 앞부분을 키워드로, 뒷부분을 설명으로 쪼개는 방식이었다.

이 방식은 실제 분석이 아니라 `문장 청킹`에 가깝다. 그래서 다음과 같이 바꾸는 것이 목표다.

| 구분 | 이전 방식 | 개선 방향 |
|---|---|---|
| 데이터 생성 | 긴 문장을 만들고 프론트에서 자름 | LLM이 처음부터 짧은 키워드/명사구 생성 |
| 화면 표시 | 키워드 + 설명 | 키워드만 표시 |
| 분석 기준 | 문자열 패턴 | 프로필·자기소개서·채용공고 비교 결과 |

현재 역량 분석 섹션은 세 그룹으로 나뉜다.

```text
나의 강점
보완할 점
지원 직무 필요 역량
```

각 그룹은 다음 JSON 필드와 연결된다.

| 화면 그룹 | LLM/백엔드 필드 | 의미 |
|---|---|---|
| 나의 강점 | `competency_gap.strengths` | 프로필과 자기소개서에 근거가 있는 강점 |
| 보완할 점 | `competency_gap.gaps` | 채용공고 대비 부족하거나 면접에서 보완 설명이 필요한 지점 |
| 지원 직무 필요 역량 | `competency_gap.required_competencies` | 채용공고와 직무 DB에서 실제로 요구되는 역량 |

### 3. 준비 항목의 부가 설명 제거

준비 항목에서는 사용자가 요청하지 않은 설명을 제거하고 다음만 남기는 방향으로 구현했다.

- 제목
- 예상 면접 질문
- 꼬리질문
- 답변 팁
- 체크박스

삭제 대상은 다음과 같다.

| 기존 표시 요소 | 제거 이유 |
|---|---|
| `준비 항목` 제목 아래 설명 | 이미 섹션 제목만으로 충분하고 화면을 복잡하게 만듦 |
| 카테고리 라벨 `역량·분야·업무` | 제목과 의미가 중복됨 |
| 카테고리 summary | 질문 준비에 직접 필요하지 않은 부가 설명 |
| 소제목 라벨 `개념 키워드` | 제목 자체가 키워드이므로 중복 |
| 소제목 아래 `why` 설명 | 질문/답변 팁보다 우선순위가 낮음 |
| evidence, study_goal | 학습 계획보다 면접 답변 준비가 목적이므로 화면에서 제외 |

## 전체 동작 흐름

분석 기능의 데이터 흐름은 다음과 같다.

```text
사용자 입력
  ↓
Frontend AnalyzeCreateView
  ↓
Django Backend /api/analyze/
  ↓
LLM Server /llm/roadmap
  ↓
GMS OpenAI Gateway 또는 Mock 응답
  ↓
Django Analysis 모델 저장
  ↓
Frontend AnalyzeResultView에서 조회 및 렌더링
```

### 1. 사용자가 분석 요청

프론트엔드에서 사용자는 다음 정보를 선택하거나 입력한다.

- 지원 직무
- 채용공고 URL 또는 직접 입력한 채용공고 내용
- 선택한 면접 유형
- 프로필 정보
- 자기소개서 정보

분석 생성 요청은 백엔드의 `/api/analyze/`로 전달된다.

### 2. 백엔드가 LLM 서버 호출

백엔드는 사용자, 직무, 회사, 채용공고 정보를 모아 LLM 서버에 전달한다.

관련 파일:

```text
backend/analysis/views.py
backend/analysis/services.py
backend/analysis/models.py
```

결과는 `Analysis` 모델에 저장된다.

중요 필드는 다음과 같다.

```python
competency_gap = models.JSONField(default=dict)
text_roadmap = models.TextField(...)
timeline_data = models.JSONField(default=list)
```

### 3. LLM 서버가 프롬프트 생성

LLM 서버는 `llm_server/roadmap_prompt.py`에서 프롬프트를 만든다.

프롬프트에는 다음 입력이 포함된다.

- 지원자 전공, 학력, 경력, 프로젝트, 자격증, 수상내역
- 자기소개서 질문/답변
- 채용공고 전문
- 회사 정보
- 직무 DB 기준 데이터
- 선택한 면접 유형

핵심 지시사항은 다음과 같다.

```text
- 채용공고 내용에서 지원 직무의 필수 역량을 먼저 추출
- 프로필, 자기소개서, 채용공고를 함께 비교
- 나의 강점, 보완할 점, 지원 직무 필요 역량을 생성
- strengths/gaps/required_competencies는 단순 청킹 금지
- 각 항목은 2~6어절의 키워드/짧은 명사구로 작성
- 준비 항목에서는 질문, 꼬리질문, 답변 팁에 필요한 내용만 작성
- summary, why, evidence, study_goal은 빈 문자열로 둠
```

### 4. LLM 응답 JSON 구조

LLM 서버가 기대하는 응답 구조는 크게 세 덩어리다.

```json
{
  "competency_gap": {
    "strengths": ["지원자의 강점 키워드"],
    "gaps": ["기업/직무 요구 대비 보완 키워드"],
    "required_competencies": ["요구 역량 키워드"]
  },
  "text_roadmap": "개인 맞춤 질문 준비 요약",
  "timeline_data": [
    {
      "category": "역량 키워드/분야/업무 대제목",
      "summary": "",
      "sources": [],
      "subtopics": [
        {
          "title": "개념 키워드 소제목",
          "why": "",
          "evidence": "",
          "study_goal": "",
          "questions": [
            {
              "question": "개인 맞춤 예상 면접 질문",
              "done": false,
              "answer_guide": "답변 팁",
              "follow_up_questions": ["꼬리질문"]
            }
          ]
        }
      ]
    }
  ]
}
```

프론트 화면은 이 JSON을 그대로 렌더링하는 것이 아니라, 필요한 필드만 골라 보여준다.

## 프론트엔드 렌더링 구조

### `AnalyzeResultView.vue`

결과 페이지의 최상위 화면이다.

주요 역할:

- `/api/analyze/:id/`로 분석 결과 조회
- 사이드바 섹션 이동 관리
- 진행률 카드 표시
- `CompetencyGap`, `RoadmapTimeline` 컴포넌트 연결

핵심 상태:

```js
const analysis = ref(null)
const loading = ref(true)
const activeSection = ref('summary')
```

분석 결과는 mount 시점에 API로 불러온다.

```js
onMounted(async () => {
  const { data } = await api.get(`/api/analyze/${route.params.id}/`)
  analysis.value = data
  initializeCompletedTasks(data.timeline_data)
})
```

상단에서 제거한 것:

```text
heroKeywords computed
hero-keywords DOM
hero-keywords CSS
```

이제 상단은 제목과 chip 중심으로 단순화된다.

### `CompetencyGap.vue`

역량 분석 카드 컴포넌트이다.

입력 prop:

```js
gap: { type: Object, default: () => ({}) }
```

내부에서는 세 배열을 computed로 분리한다.

```js
const strengths = computed(() => props.gap.strengths || [])
const gaps = computed(() => props.gap.gaps || [])
const requiredCompetencies = computed(() => props.gap.required_competencies || [])
```

화면 그룹:

```vue
<div class="gap-title">나의 강점</div>
<div class="gap-title">보완할 점</div>
<div class="gap-title">지원 직무 필요 역량</div>
```

기술적으로 중요한 점은 `splitKeywordDescription` 방식이 제거되었다는 것이다.

이전에는 프론트에서 문자열을 억지로 쪼갰다.

```js
splitKeywordDescription(text)
```

개선 후에는 LLM이 키워드를 제대로 만들어준다는 전제하에 화면은 키워드만 표시한다.

```js
function getKeywordValue(item, raw) {
  if (item && typeof item === 'object') {
    return String(item.keyword || item.title || item.name || item.concept || '')
      .replace(/^\(Mock\)\s*/, '')
      .trim()
  }
  return raw.replace(/^\(Mock\)\s*/, '').replace(/\s+/g, ' ').trim()
}
```

이 함수는 문자열뿐 아니라 객체 형태 응답도 어느 정도 허용한다.

예를 들어 다음 두 형태 모두 처리할 수 있다.

```json
"로봇 제어 SW"
```

```json
{ "keyword": "로봇 제어 SW" }
```

### `RoadmapTimeline.vue`

준비 항목 전체 리스트를 렌더링한다.

역할:

- `timelineData`를 순회
- 각 category를 `RoadmapCategoryCard`로 전달
- 현재 진행 중인 category 계산
- 체크 이벤트를 상위로 전달

현재 category 계산 방식:

```js
const currentCategoryIdx = computed(() => {
  for (let categoryIdx = 0; categoryIdx < props.timelineData.length; categoryIdx++) {
    const category = props.timelineData[categoryIdx]
    const hasIncomplete = category.subtopics.some((subtopic, subtopicIdx) => {
      return subtopic.questions.some((_, questionIdx) => {
        return !props.completedTasks[`${categoryIdx}-${subtopicIdx}-${questionIdx}`]
      })
    })
    if (hasIncomplete) return categoryIdx
  }
  return Math.max(0, props.timelineData.length - 1)
})
```

즉, 아직 체크하지 않은 질문이 있는 첫 번째 카테고리를 현재 항목으로 본다.

### `RoadmapCategoryCard.vue`

준비 항목의 큰 묶음 카드이다.

표시하는 것:

- category 제목
- 진행 상태
- 하위 subtopic 목록

표시하지 않는 것:

- category label
- summary 설명
- sources 출처

진행 상태는 질문 체크 개수를 기준으로 계산한다.

```js
const completedCount = computed(() => {
  return props.category.subtopics.reduce((count, subtopic, subtopicIdx) => {
    return count + subtopic.questions.filter((_, questionIdx) =>
      isQuestionCompleted(subtopicIdx, questionIdx)
    ).length
  }, 0)
})
```

상태 텍스트 규칙:

| 조건 | 표시 |
|---|---|
| 질문이 없음 | 대기 |
| 모든 질문 완료 | 완료 |
| 일부 완료 또는 현재 카테고리 | `완료수/전체수` |
| 시작 전 | 대기 |

### `RoadmapSubtopicCard.vue`

실제 면접 질문을 보여주는 카드이다.

표시하는 것:

- 소제목
- 질문 번호
- 예상 질문
- 꼬리질문
- 답변 팁
- 체크박스

표시하지 않는 것:

- `개념 키워드` 라벨
- `why` 설명
- evidence
- study_goal

질문 하나의 구조는 다음과 같다.

```json
{
  "question": "1번 프로젝트에서 역기구학을 어떻게 사용했나요?",
  "done": false,
  "answer_guide": "프로젝트 목표, 목표 위치 계산, 관절각 산출 순서로 답변하세요.",
  "follow_up_questions": ["관절 제한은 어느 단계에서 반영했나요?"]
}
```

렌더링은 다음 기준으로 이루어진다.

```vue
<p class="question-text">{{ questionItem.question }}</p>

<div v-if="questionItem.follow_up_questions.length">
  <span>꼬리질문</span>
</div>

<div v-if="questionItem.answer_guide">
  <span>답변 팁</span>
</div>
```

## 진행률과 체크 상태 동작 원리

진행률은 `frontend/src/composables/useRoadmapProgress.js`에서 관리한다.

### 1. timeline 데이터 정규화

LLM 응답은 항상 완벽히 같은 구조로 오지 않을 수 있다.

그래서 `normalizeRoadmap` 함수가 두 형태를 처리한다.

| 형태 | 처리 함수 |
|---|---|
| category/subtopics 구조 | `normalizeCategoryItem` |
| 예전 week/tasks 구조 | `normalizeLegacyItem` |

```js
function normalizeRoadmap(timeline = []) {
  return timeline.map((item, itemIdx) => {
    if (Array.isArray(item.subtopics)) return normalizeCategoryItem(item, itemIdx)
    return normalizeLegacyItem(item, itemIdx)
  })
}
```

### 2. 질문 개수 계산

전체 질문 개수는 모든 category, subtopic, question을 합산한다.

```js
const totalTaskCount = computed(() => {
  return roadmapItems.value.reduce((count, category) => {
    return count + category.subtopics.reduce(
      (subCount, subtopic) => subCount + subtopic.questions.length,
      0
    )
  }, 0)
})
```

### 3. 완료 개수와 퍼센트 계산

체크된 질문만 완료로 본다.

```js
const completedTaskCount = computed(() =>
  Object.values(completedTasks.value).filter(Boolean).length
)
```

진행률은 다음 공식이다.

```text
완료 질문 수 / 전체 질문 수 * 100
```

```js
const progressPercent = computed(() => {
  if (totalTaskCount.value === 0) return 0
  return Math.round((completedTaskCount.value / totalTaskCount.value) * 100)
})
```

### 4. localStorage 저장

체크 상태는 새로고침해도 유지되도록 localStorage에 저장한다.

저장 key:

```js
`roadmap-progress:${analysis.id}`
```

즉 분석 결과 ID별로 체크 상태가 따로 저장된다.

```js
const storageKey = computed(() => {
  const id = analysis.value?.id
  return id ? `roadmap-progress:${id}` : ''
})
```

## LLM 서버의 Mock 응답 역할

`llm_server/roadmap_mock.py`는 `GMS_KEY`가 없을 때 실제 LLM 대신 반환되는 개발용 응답이다.

동작 조건:

```python
if not GMS_KEY:
    return MOCK_ROADMAP_RESPONSE
```

Mock 응답도 실제 프롬프트 방향과 맞아야 하므로 다음을 반영했다.

- strengths/gaps/required_competencies는 짧은 키워드 중심
- category summary는 빈 문자열
- subtopic why는 빈 문자열
- 질문, 꼬리질문, 답변 팁은 유지

이렇게 해야 로컬 개발 환경에서도 실제 결과 화면과 비슷한 UI를 확인할 수 있다.

## 기존 분석 결과와 신규 분석 결과의 차이

중요한 점은 프롬프트 수정은 `새로 생성되는 분석 결과`에 적용된다는 것이다.

이미 DB에 저장된 기존 분석 결과는 과거 LLM 응답을 가지고 있을 수 있다.

그래서 프론트엔드에서는 다음 두 방식을 함께 적용한다.

1. 기존 데이터에 summary, why가 있어도 화면에서 보여주지 않음
2. 신규 데이터는 LLM에게 summary, why를 빈 문자열로 만들도록 요청

이렇게 하면 기존 결과와 신규 결과 모두 화면 정책은 동일하게 유지된다.

## 기술적으로 기억할 점

### 1. 표시 정책은 프론트엔드가 최종 결정한다

LLM이 `summary`나 `why`를 만들어도 프론트에서 렌더링하지 않으면 화면에는 보이지 않는다.

즉, 화면에 무엇을 보여줄지는 최종적으로 Vue 컴포넌트가 결정한다.

### 2. 생성 품질은 LLM 프롬프트가 좌우한다

역량 분석의 품질은 프론트에서 문자열을 자르는 방식보다 LLM이 처음부터 올바른 키워드를 만들도록 지시하는 것이 중요하다.

그래서 프롬프트에 다음 제약을 넣었다.

```text
단순히 긴 문장을 앞부분만 자른 청킹 결과가 아니어야 합니다.
각 항목은 입력 근거를 종합해 새로 붙인 2~6어절의 키워드/짧은 명사구로 작성하세요.
```

### 3. 정규화 계층이 있어야 UI가 안정적이다

LLM 응답은 가끔 필드가 빠지거나 배열 구조가 달라질 수 있다.

`useRoadmapProgress.js`의 정규화 함수는 이런 변화에도 UI가 완전히 깨지지 않게 하는 완충 계층이다.

### 4. 체크 상태 key는 위치 기반이다

체크 상태는 다음 형식의 key로 저장된다.

```text
categoryIdx-subtopicIdx-questionIdx
```

예시:

```text
0-1-2
```

의미:

```text
첫 번째 카테고리의 두 번째 소제목의 세 번째 질문
```

따라서 같은 분석 결과에서 질문 순서가 바뀌면 저장된 체크 상태와 질문의 매칭이 달라질 수 있다. 현재는 분석 결과가 생성된 뒤 timeline 구조가 바뀌지 않는다는 전제에서 문제가 없다.

## 최종 정리

이번 개선의 핵심은 `많이 보여주는 화면`에서 `면접 준비에 필요한 것만 보여주는 화면`으로 바꾸는 것이다.

- 상단에서는 중복 역량 요약을 제거했다.
- 역량 분석은 프로필·자기소개서·채용공고를 비교한 키워드 중심으로 정리한다.
- 준비 항목에서는 부가 설명을 제거하고 질문, 꼬리질문, 답변 팁만 남긴다.
- 진행률은 질문 체크 상태를 기준으로 계산하고, localStorage에 저장한다.
- 신규 분석 결과 품질은 LLM 프롬프트에서 단순 청킹을 금지하고 입력 근거 기반 키워드 생성을 요구함으로써 개선한다.
