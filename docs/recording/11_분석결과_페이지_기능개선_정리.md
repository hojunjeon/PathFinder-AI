# 분석 결과 페이지 기능 개선 정리

## 문서 목적

이 문서는 분석 결과 페이지에서 최근 진행한 기능 개선 내용을 나중에 다시 이해할 수 있도록 정리한 기술 문서이다.

정리 범위는 다음과 같다.

1. 왼쪽 `섹션 이동` 내비게이션의 스크롤 동작 개선
2. `역량 분석`과 `직무 역량 매칭도`의 키워드 중심 표시 정책 개선
3. `준비 항목`을 대제목-소제목-질문 체크리스트 구조로 개편
4. LLM 서버가 위 화면 구조에 맞는 JSON을 생성하도록 프롬프트와 mock 응답을 조정한 내용
5. 각 기능이 프론트엔드와 LLM 서버에서 어떤 원리로 동작하는지

---

## 최종 사용자 경험 기준

분석 결과 페이지는 다음 네 구역으로 구성된다.

```text
분석 요약
역량 분석
직무 매칭도
준비 항목
```

왼쪽 사이드바의 `섹션 이동`에서 각 항목을 클릭하면 해당 구역으로 이동한다.

각 구역의 역할은 다음과 같다.

| 구역 | 역할 |
|---|---|
| 분석 요약 | 회사명, 면접 유형, 전체 준비 진행률을 빠르게 확인 |
| 역량 분석 | 프로필·자기소개서·채용공고를 바탕으로 나의 강점, 보완할 점, 지원 직무 필요 역량을 키워드로 확인 |
| 직무 매칭도 | 강점/요구/보완 역량을 짧은 키워드와 막대 점수로 확인 |
| 준비 항목 | 역량/분야/업무별 예상 질문, 꼬리질문, 답변 팁을 체크리스트로 관리 |

핵심 방향은 `긴 설명형 로드맵`이 아니라 `면접 직전에 바로 확인 가능한 질문 준비표`이다.

## 1. 섹션 이동 스크롤 개선

### 문제

기존에는 왼쪽 사이드바의 `분석 요약`, `역량 분석`, `직무 매칭도`, `준비 항목`을 클릭하면 브라우저 기본 anchor 이동만 사용했다.

```vue
<a :href="`#${section.id}`">
```

이 방식은 다음 문제가 있었다.

- sticky header와 화면 여백 때문에 섹션 제목이 화면 상단에 정확히 맞지 않음
- 클릭 후 실제로 보고 싶은 카드의 시작 위치보다 살짝 위/아래로 이동함
- 현재 활성 섹션 표시가 사용자가 보는 위치와 어긋날 수 있음

### 변경 내용

`AnalyzeResultView.vue`에서 클릭 이벤트를 직접 처리하도록 변경했다.

```vue
<a
  :href="`#${section.id}`"
  @click.prevent="scrollToSection(section.id)"
>
```

`준비 항목 보기` 버튼도 같은 방식으로 변경했다.

```vue
<a href="#roadmap" class="content-btn" @click.prevent="scrollToSection('roadmap')">
  준비 항목 보기
</a>
```

### 동작 원리

`scrollToSection(id)` 함수는 다음 순서로 동작한다.

1. `document.getElementById(id)`로 이동 대상 섹션을 찾는다.
2. `activeSection` 값을 먼저 갱신해 사이드바 활성 상태를 즉시 반영한다.
3. `el.scrollIntoView({ behavior: 'smooth', block: 'start' })`로 부드럽게 이동한다.
4. `history.replaceState(null, '', '#id')`로 주소 hash를 갱신하되 브라우저 히스토리를 불필요하게 추가하지 않는다.

관련 코드 위치:

- `frontend/src/views/AnalyzeResultView.vue`
  - `scrollToSection(id)`
  - `handleScroll()`
  - `.result-hero { scroll-margin-top: 64px; }`
  - `.section-card { scroll-margin-top: 64px; }`

### 왜 `scroll-margin-top`을 같이 쓰는가

`scrollIntoView({ block: 'start' })`는 대상 요소의 시작점을 viewport 상단에 맞춘다.

하지만 이 프로젝트는 상단 고정 영역과 카드 여백이 있으므로, 요소가 완전히 위에 붙으면 제목이 답답하게 보이거나 일부가 가려질 수 있다.

그래서 이동 대상 요소에 다음 CSS를 둔다.

```css
scroll-margin-top: 64px;
```

이 값 덕분에 anchor 이동 시 섹션 시작점이 화면 상단에서 64px 아래에 위치한다.

## 2. 현재 활성 섹션 계산 방식 개선

### 기존 방식

기존에는 `window.scrollY >= el.offsetTop - 180` 조건으로 현재 섹션을 판단했다.

```js
if (el && window.scrollY >= el.offsetTop - 180) {
  current = id
}
```

이 방식은 문서 전체 좌표 기준으로 단순 비교하므로, 실제 viewport에서 어느 섹션이 가장 가까운지와 다를 수 있다.

### 변경 방식

현재 구현은 `getBoundingClientRect().top`을 사용해 viewport 기준 거리로 판단한다.

동작 개념은 다음과 같다.

```js
const anchorOffset = 72
const distance = Math.abs(el.getBoundingClientRect().top - anchorOffset)
```

즉, 각 섹션의 top 위치가 화면 상단 기준점인 `72px`에 얼마나 가까운지 계산한다.

그리고 사용자가 문서 맨 아래에 거의 도달하면 마지막 섹션인 `준비 항목`을 활성화한다.

```js
const scrollBottom = window.innerHeight + window.scrollY
const documentHeight = document.documentElement.scrollHeight
if (documentHeight - scrollBottom < 8) {
  current = lastSection.id
}
```

이렇게 한 이유는 페이지 마지막 섹션은 콘텐츠 높이에 따라 기준점까지 올라오지 못할 수 있기 때문이다.

## 3. 역량 분석 표시 정책 개선

### 변경 전

`CompetencyGap.vue`는 LLM이 내려준 긴 문장을 프론트에서 임의로 잘라 `키워드 + 설명`처럼 표시했다.

예를 들어 다음 문자열이 있으면,

```text
로봇 제어 SW 개발 역량 - 프로젝트 경험과 채용공고 요구가 연결됩니다
```

프론트에서 하이픈, 콜론, 괄호, 공백 등을 기준으로 나누었다.

이 방식은 실제 분석이라기보다 문자열 패턴 처리에 가까웠다.

### 변경 후

현재는 화면에서 설명 문단을 제거하고 키워드만 보여준다.

구역 이름도 사용자가 이해하기 쉽게 변경했다.

| 기존 | 변경 |
|---|---|
| 강점 키워드 | 나의 강점 |
| 보완 키워드 | 보완할 점 |
| 요구 역량 키워드 | 지원 직무 필요 역량 |

상단 보조 문구도 다음처럼 변경했다.

```text
프로필·자기소개서·채용공고 기반
```

### 기술적 동작 원리

`toKeywordItems(items)`가 `strengths`, `gaps`, `required_competencies` 배열을 화면 표시용 키워드 배열로 정규화한다.

```js
const strengthKeywords = computed(() => toKeywordItems(strengths.value))
const gapKeywords = computed(() => toKeywordItems(gaps.value))
const requiredKeywords = computed(() => toKeywordItems(requiredCompetencies.value))
```

정규화 함수는 문자열과 객체형 응답을 모두 받을 수 있게 처리한다.

```js
function getRawValue(item) {
  if (item && typeof item === 'object') {
    return String(item.keyword || item.title || item.name || item.concept || '').trim()
  }
  return String(item || '').trim()
}
```

이렇게 한 이유는 LLM 응답 형식이 나중에 다음처럼 확장될 수 있기 때문이다.

```json
{
  "keyword": "로봇 제어 SW",
  "reason": "..."
}
```

또는 현재처럼 단순 문자열 배열일 수도 있다.

```json
["로봇 제어 SW", "산업용 통신"]
```

프론트는 두 경우를 모두 안전하게 받아 키워드만 표시한다.

관련 파일:

- `frontend/src/components/result/CompetencyGap.vue`

## 4. 직무 역량 매칭도 키워드형 표시

### 문제

`직무 역량 매칭도`는 기존에 `competency_gap`에 들어온 문자열 전체를 막대 이름으로 사용했다.

LLM 응답이 긴 문장으로 내려오면 막대 왼쪽 라벨이 길어져 가독성이 떨어졌다.

### 변경 내용

`AnalyzeResultView.vue`에서 `toKeywordLabel(value)` 함수를 추가해 긴 문장을 짧은 키워드로 압축한다.

```js
function toKeywordLabel(value) {
  const text = String(value || '')
    .replace(/^\(Mock\)\s*/, '')
    .replace(/\s+/g, ' ')
    .trim()

  if (!text) return ''

  const parenMatch = text.match(/^(.+?)\s*\(.+\)$/)
  if (parenMatch) return compactKeyword(parenMatch[1])

  const separatorMatch = text.match(/^(.{2,32}?)(?:\s*[:：]\s+|\s+-\s+|\s+대비\s+|\s+보완\s+)(.+)$/)
  if (separatorMatch) return compactKeyword(separatorMatch[1])

  return compactKeyword(text)
}
```

`compactKeyword(text)`는 문자열이 너무 길면 공백, `/`, `·`, 쉼표 등을 기준으로 앞쪽 핵심 토큰만 남긴다.

```js
return tokens.slice(0, 3).join(' ')
```

### 점수 생성 방식

현재 점수는 실제 채점 모델이 아니라, 같은 키워드에 대해 항상 같은 값을 만들기 위한 deterministic score이다.

```js
function getDeterministicScore(name, minVal, maxVal) {
  let hash = 0
  for (let i = 0; i < name.length; i++) {
    hash = name.charCodeAt(i) + ((hash << 5) - hash)
  }
  const range = maxVal - minVal
  return minVal + Math.abs(hash % range)
}
```

동작 원리는 다음과 같다.

1. 키워드 문자열의 각 글자 코드를 이용해 hash 값을 만든다.
2. hash를 지정 범위로 나눈 나머지를 사용한다.
3. 강점/요구/보완 유형별로 다른 점수 범위를 적용한다.

| 유형 | 점수 범위 | 의미 |
|---|---:|---|
| 강점 | 75~95 | 이미 근거가 있는 역량 |
| 요구 | 45~70 | 직무상 필요하지만 더 확인이 필요한 역량 |
| 보완 | 15~40 | 면접 전 답변 보강이 필요한 역량 |

이 점수는 사용자의 실제 실력을 정밀 측정하는 값이라기보다, 면접 준비 우선순위를 시각적으로 구분하기 위한 UI 값이다.

관련 파일:

- `frontend/src/views/AnalyzeResultView.vue`

## 5. 준비 항목 구조 개편

### 목표 구조

사용자 요청에 맞춰 준비 항목은 다음 계층으로 정리했다.

```text
대제목: 역량 키워드 / 분야 / 업무
  소제목: 개념 키워드
    질문 1
      꼬리질문
      답변 팁
      체크박스
    질문 2
      꼬리질문
      답변 팁
      체크박스
```

예시 JSON 구조는 다음과 같다.

```json
{
  "category": "로보틱스",
  "summary": "",
  "sources": [],
  "subtopics": [
    {
      "title": "역기구학",
      "why": "",
      "evidence": "",
      "study_goal": "",
      "questions": [
        {
          "question": "1번 프로젝트에서 역기구학을 어떻게 사용했나요?",
          "done": false,
          "answer_guide": "프로젝트 목표, 목표 위치 계산, 관절각 산출, 채용공고의 제어 소프트웨어 요구와의 연결 순서로 답변하세요.",
          "follow_up_questions": ["관절 제한은 어느 단계에서 반영했나요?"]
        }
      ]
    }
  ]
}
```

### 화면 표시 변경

현재 화면에서는 `summary`, `why` 같은 부가 설명을 직접 표시하지 않는다.

화면에는 다음만 집중해서 보여준다.

- 대제목: `category`
- 소제목: `subtopic.title`
- 질문: `questionItem.question`
- 꼬리질문: `questionItem.follow_up_questions`
- 답변 팁: `questionItem.answer_guide`
- 체크박스: 질문별 완료 상태

관련 파일:

- `frontend/src/components/result/RoadmapCategoryCard.vue`
- `frontend/src/components/result/RoadmapSubtopicCard.vue`
- `frontend/src/components/result/RoadmapTimeline.vue`

## 6. 준비 항목 진행률 계산 원리

### 기존 개념

이전에는 진행률을 subtopic 단위로 계산했다.

즉, 개념 키워드 하나를 체크하면 해당 항목 전체가 완료되는 방식이었다.

### 변경 개념

현재는 질문 단위로 진행률을 계산한다.

즉, 하나의 개념 키워드 아래에 질문이 여러 개 있으면 질문 각각이 개별 체크 항목이다.

```text
로보틱스 / 역기구학 / 질문 1 -> 체크 1개
로보틱스 / 역기구학 / 질문 2 -> 체크 1개
로보틱스 / 모션 플래닝 / 질문 1 -> 체크 1개
```

### key 구조

체크 상태는 `completedTasks` 객체에 저장된다.

각 질문은 다음 key를 가진다.

```text
categoryIdx-subtopicIdx-questionIdx
```

예시:

```text
0-0-0
0-0-1
0-1-0
```

### 진행률 계산

`useRoadmapProgress.js`에서 전체 질문 수를 계산한다.

```js
const totalTaskCount = computed(() => {
  return roadmapItems.value.reduce((count, category) => {
    return count + category.subtopics.reduce((subCount, subtopic) => subCount + subtopic.questions.length, 0)
  }, 0)
})
```

완료 수는 `completedTasks`에서 `true`인 값의 개수이다.

```js
const completedTaskCount = computed(() => Object.values(completedTasks.value).filter(Boolean).length)
```

진행률은 다음 공식으로 계산한다.

```text
진행률 = 완료 질문 수 / 전체 질문 수 * 100
```

```js
const progressPercent = computed(() => {
  if (totalTaskCount.value === 0) return 0
  return Math.round((completedTaskCount.value / totalTaskCount.value) * 100)
})
```

### 체크 이벤트 흐름

질문 체크박스를 클릭하면 이벤트는 다음 순서로 올라간다.

```text
RoadmapSubtopicCard
  -> toggle-question { subtopicIdx, questionIdx }
RoadmapCategoryCard
  -> toggle-question { subtopicIdx, questionIdx }
RoadmapTimeline
  -> toggle-task { categoryIdx, subtopicIdx, questionIdx }
AnalyzeResultView
  -> toggleTask(...)
useRoadmapProgress
  -> completedTasks 갱신
```

실제 이벤트 코드 흐름은 다음과 같다.

```vue
@change="$emit('toggle-question', { subtopicIdx, questionIdx })"
```

```vue
@toggle-question="payload => $emit('toggle-task', { categoryIdx, ...payload })"
```

```vue
<RoadmapTimeline
  :timeline-data="roadmapItems"
  :completed-tasks="completedTasks"
  @toggle-task="toggleTask"
/>
```

## 7. 로드맵 데이터 정규화 원리

LLM 응답이 항상 새 구조로만 온다고 가정하면 위험하다.

기존 분석 결과에는 다음처럼 예전 구조가 저장되어 있을 수 있다.

```json
{
  "title": "1주차",
  "tasks": ["프로젝트 정리", "기술 질문 준비"]
}
```

그래서 `useRoadmapProgress.js`는 `normalizeRoadmap()`으로 데이터를 화면 표시용 구조로 변환한다.

### 새 구조 처리

새 구조는 `subtopics` 배열을 가진다.

```js
if (Array.isArray(item.subtopics)) return normalizeCategoryItem(item, itemIdx)
```

`normalizeCategoryItem()`은 각 subtopic을 `normalizeSubtopic()`으로 변환한다.

`normalizeSubtopic()`은 다음을 보장한다.

- `title`은 항상 존재
- `questions` 배열은 항상 존재
- `question`, `answer_guide`, `follow_up_questions`, `done`을 질문 단위로 정규화
- 기존에 subtopic 레벨에 있던 `question`, `answer_guide`, `follow_up_questions`도 새 `questions` 배열로 감싸서 표시 가능

### 예전 구조 처리

예전 구조는 `tasks` 배열을 질문 1개짜리 subtopic으로 변환한다.

```js
questions: [{
  question: String(task),
  answer_guide: '',
  follow_up_questions: [],
  done: false,
  hasDone: false,
}]
```

덕분에 기존 저장 데이터도 새 UI에서 깨지지 않는다.

관련 파일:

- `frontend/src/composables/useRoadmapProgress.js`

## 8. LLM 프롬프트 변경

### 변경 이유

프론트에서 긴 문장을 자르는 방식만으로는 사용자가 원하는 결과를 만들기 어렵다.

특히 다음 요구사항은 데이터 생성 단계에서부터 반영되어야 한다.

- 강점/보완점/필요 역량은 긴 설명이 아니라 키워드로 생성
- 준비 항목은 대제목-소제목-질문 배열 구조로 생성
- 질문마다 체크 상태, 꼬리질문, 답변 팁이 있어야 함
- 답변 팁은 프로필·자기소개서·채용공고를 어떻게 연결할지 안내해야 함

### 주요 지시 변경

`llm_server/roadmap_prompt.py`의 분석 지시를 다음 방향으로 강화했다.

1. 채용공고에서 필수 역량을 먼저 추출
2. 사용자 프로필, 자기소개서, 채용공고를 함께 비교
3. `strengths/gaps/required_competencies`는 단순 문장 청킹이 아니라 새로 붙인 2~6어절 키워드/명사구로 작성
4. `timeline_data.category`는 역량 키워드/분야/업무 대제목으로 작성
5. `subtopics.title`은 개념 키워드 소제목으로 작성
6. `questions` 배열 안에 질문별 `question`, `done`, `answer_guide`, `follow_up_questions`를 작성
7. 화면에는 부가 설명을 보여주지 않으므로 `summary`, `why`, `evidence`, `study_goal`은 빈 문자열로 유지

### 출력 JSON 형식 변경

핵심 변경은 `subtopics` 안에 직접 `question`을 두는 방식에서 `questions` 배열을 두는 방식이다.

변경 후:

```json
"subtopics": [
  {
    "title": "개념 키워드 소제목",
    "why": "",
    "evidence": "",
    "study_goal": "",
    "questions": [
      {
        "question": "개인 맞춤 예상 면접 질문",
        "done": false,
        "answer_guide": "답변 팁",
        "follow_up_questions": ["꼬리질문"]
      }
    ]
  }
]
```

관련 파일:

- `llm_server/roadmap_prompt.py`

## 9. Mock 응답 변경

`GMS_KEY`가 없으면 LLM 서버는 mock 응답을 사용한다.

따라서 실제 GMS 호출이 없어도 새 UI 구조를 확인할 수 있도록 `llm_server/roadmap_mock.py`도 새 구조에 맞게 수정했다.

변경된 mock 응답 특징:

- `strengths`, `gaps`, `required_competencies`가 키워드/짧은 명사구 중심
- `timeline_data[].summary`는 빈 문자열
- `subtopics[].why`는 빈 문자열
- 각 subtopic은 `questions` 배열을 가짐
- 각 question은 `question`, `done`, `answer_guide`, `follow_up_questions`를 가짐

LLM 프롬프트 내용이 바뀌었기 때문에 기존 테스트도 새 정책을 확인하도록 조정했다.

관련 파일:

## 12. 전체 데이터 흐름

분석 결과 페이지의 전체 데이터 흐름은 다음과 같다.

```text
사용자 입력
  -> backend analysis API
  -> llm_server /llm/roadmap
  -> roadmap_prompt.py가 프롬프트 생성
  -> GMS/OpenAI 또는 mock 응답
  -> backend Analysis 모델에 competency_gap, text_roadmap, timeline_data 저장
  -> frontend AnalyzeResultView가 /api/analyze/:id/ 호출
  -> useRoadmapProgress가 timeline_data 정규화
  -> CompetencyGap / RoadmapTimeline 컴포넌트가 화면 렌더링
  -> 사용자가 질문 체크
  -> localStorage에 진행률 상태 저장
```

진행률 저장 key는 분석 ID를 기준으로 만들어진다.

```text
roadmap-progress:{analysis.id}
```

이 구조 덕분에 분석 결과별로 체크 상태가 분리된다.

## 13. 주의할 점

### 1. 이전 분석 결과와의 호환성

기존 DB에 저장된 `timeline_data`는 새 `questions` 배열이 없을 수 있다.

이를 위해 `useRoadmapProgress.js`에서 legacy 데이터를 새 구조로 변환한다.

따라서 기존 분석 결과도 최소한 질문 1개짜리 항목으로 표시된다.

### 2. 점수는 실제 평가 모델이 아니다

`직무 역량 매칭도` 점수는 키워드 hash 기반 deterministic UI 값이다.

같은 키워드에 같은 점수를 부여해 일관성을 주지만, 실제 역량 점수라고 해석하면 안 된다.

### 3. LLM 응답 품질은 프롬프트에 의존한다

프론트는 표시 구조를 안정화하지만, 질문과 답변 팁의 품질은 LLM 응답에 영향을 받는다.

그래서 `roadmap_prompt.py`에서 다음 지시를 명확히 했다.

- 입력 근거를 종합해 키워드를 새로 붙일 것
- 질문은 개인 프로필·자기소개서·채용공고 기반으로 작성할 것
- 답변 팁은 어떤 순서와 관점으로 답변할지 제안할 것

## 결론

이번 변경의 핵심은 분석 결과 페이지를 `설명 많은 분석 문서`에서 `면접 준비 실행 화면`으로 바꾸는 것이다.

기술적으로는 다음 세 가지가 가장 중요하다.

1. 스크롤 이동은 anchor 기본 동작이 아니라 `scrollIntoView + scroll-margin-top + activeSection 계산`으로 제어한다.
2. 역량/매칭도 표시는 긴 문장을 프론트에서 억지로 나누는 방식보다 LLM이 키워드를 생성하고 프론트는 표시만 담당하는 방향으로 바꾼다.
3. 준비 항목 진행률은 카테고리나 소제목이 아니라 실제 사용자가 준비해야 할 `질문 단위`로 계산한다.
