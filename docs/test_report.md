# PathFinder AI E2E 테스트 및 기능 분석 결과 보고서

본 보고서는 동일한 로컬 환경에서 백엔드(Django), 프론트엔드(Vue 3), LLM 서버(FastAPI) 개발 서버를 실행한 상태에서 Playwright를 활용해 수행한 E2E 테스트 결과 및 요구사항 분석 내용을 종합적으로 정리한 문서입니다.

## 2. 요구사항 대비 분석 및 문제점 (Gaps & Bugs)

서브에이전트들의 E2E 테스트 실행 결과와 프론트엔드/백엔드/LLM 서버 코드 분석을 종합한 결과, 다음과 같은 버그 및 요구사항 미비점이 발견되었습니다.

### ① 프로필 저장 예외 처리 버그 (UX 버그)
* **현황**: `ProfileView.vue` 코드의 `save` 함수 내에서 백엔드 API 호출(`api.put('/api/profile/')`)이 실패(네트워크 장애, 토큰 만료 등)하더라도, catch 블록에서 에러 메시지를 노출하지 않고 단순 통과시키며 즉시 `saved.value = true` 처리를 수행합니다.
* **영향**: 사용자는 프로필 저장이 실제로 실패했음에도 불구하고 화면상으로 "✓ 저장되었습니다."라는 허위 성공 알림을 받게 되어, 입력 데이터 유실로 이어질 수 있습니다.

### ② LLM 로드맵 프롬프트 내 주요 데이터 누락 (기능적 누락)
* **현황**: 
  * `요구사항.md`에 의하면 **"서류 접수 시 제출한 자소서 등을 입력"**하고 **"사용자 정보를 종합해서 현재 수준을 파악"**해야 합니다.
  * 백엔드 Django(`build_llm_payload`)에서는 사용자가 제출한 자소서 데이터를 `'자소서'` 필드에 담아 LLM 서버로 전달합니다.
  * 그러나 LLM 서버(`llm_server/main.py`)의 `_build_prompt` 템플릿에는 수신한 `자소서` 데이터를 주입하는 로직이 빠져 있어, AI가 자소서 내용을 전혀 반영하지 않고 로드맵을 작성합니다.
  * 추가적으로, 사용자 프로필의 `수상내역(awards)` 필드 또한 LLM 프롬프트 구성 시 완전히 배제되어 있습니다.

### ③ 채용공고 스크래핑(Scraping)의 기술적 한계 (기능적 취약점)
* **현황**: 백엔드 Django(`backend/analysis/services.py`의 `fetch_job_posting_text`)에서 채용공고 텍스트를 추출할 때 단순 `httpx.get`을 통한 정적 HTML 파싱을 시도합니다.
* **영향**: 최신 채용 플랫폼(쿠팡, 원티드 등)은 SPA 방식으로 렌더링되거나 Cloudflare 등 봇 차단 솔루션을 적용하기 때문에, 정적 HTML 파싱 시 본문을 가져오지 못하고 URL 주소만 LLM으로 넘어가게 되어 개인화 로드맵의 품질이 심각하게 저하됩니다.

### ④ 요구사항 범위 불일치 (기획 스코프 불일치)
* **현황**: 구현된 대시보드 페이지(`/dashboard`)는 채용시장 경쟁률 분석 기능(Chart.js 기반 4개 차트 렌더링, PNG 다운로드 등)을 제공하며 E2E 테스트도 존재합니다. 하지만 이 기능은 `요구사항.md`에 언급되지 않은 기획 외 기능(Out-of-scope)입니다.

### ⑤ GMS_KEY 미설정 시 서비스 불능
* **현황**: 로컬 개발 서버 작동 시 `GMS_KEY` 환경 변수가 존재하지 않으면 LLM 서버가 `503 Service Unavailable`을 반환합니다. E2E 테스트는 API 모킹(`page.route`)을 활용해 이 문제를 우회하고 있으나, 실제 로컬 수동 테스트 시에는 서비스 이용이 원천 차단됩니다.
