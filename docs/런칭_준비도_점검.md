# PathFinder AI 런칭 준비도 점검

작성일: 2026-06-21

## 결론

현재 상태로는 **런칭 보류**가 맞습니다.
핵심 사용자 흐름(로그인 → 기업/직무 선택 → 분석 생성 → 결과 확인)은 데모 수준으로는 보이지만, 실제 서비스 런칭 관점에서는 **배포 설정, 데이터 준비, 보안, 운영 경로**가 아직 부족합니다.

## 런칭 전 반드시 막아야 하는 이슈 (Blocker)

| 구분 | 내용 | 근거 |
|---|---|---|
| 배포 | 프론트 API 주소가 `http://localhost:8080`으로 하드코딩되어 있어 배포 후에도 사용자 브라우저가 자기 PC의 localhost를 호출하게 됩니다. | `frontend/src/api/index.js:4`, `frontend/src/api/index.js:23` |
| 데이터 | 현재 DB에 `companies`와 `jobs`가 **0건**이라서 핵심 기능인 기업 선택/직무 선택/분석 생성이 사실상 동작하지 않습니다. 시드 커맨드는 있지만 자동 실행되지 않습니다. | `backend/db.sqlite3` 테이블 상태, `backend/companies/management/commands/seed_companies.py`, `scripts/run-dev-servers.ps1:275-280` |
| LLM 신뢰성 | `GMS_KEY`가 없어도 LLM 서버가 **가짜(mock) 로드맵**을 성공 응답으로 반환하고, 백엔드는 그 결과를 `DONE`으로 저장합니다. 운영에서 이 상태로 나가면 허위 분석 결과를 사용자에게 줄 수 있습니다. | `llm_server/main.py:151-165`, `backend/analysis/views.py:40-55` |

## 지금 가장 먼저 채워야 할 런칭 체크리스트

1. **프론트 API URL 환경변수화**
   - `localhost` 하드코딩 제거
   - 배포 환경별 API origin 분리

2. **운영용 데이터 적재 절차 확정**
   - `seed_companies` 실행 기준 정의
   - 최소 운영 데이터셋 적재
   - 배포 후 빈 DB가 뜨지 않도록 체크

3. **LLM 운영 모드 정리**
   - 운영에서는 mock 성공 응답 금지
   - `GMS_KEY` 없으면 시작 실패 또는 명시적 장애 처리
   - malformed LLM 응답은 `DONE` 저장 금지

4. **채용공고 수집 보안 보강**
   - SSRF 차단 강화
   - 허용 도메인 기반 수집 또는 별도 수집 계층 검토

5. **배포 경로/런북 작성**
   - 프론트 정적 배포 방식
   - Django/LLM 서버 실행 방식
   - 환경변수 목록
   - health check / 로그 / 장애 대응 절차

6. **운영 DB 전환**
   - PostgreSQL 등으로 전환
   - 마이그레이션/백업/복구 절차 문서화

7. **실통합 테스트 추가**
   - 실제 Django + LLM(mock이 아닌 운영 모드 또는 통제된 stub) + 브라우저까지 연결한 smoke test
   - 최소 1개는 “회원가입 → 프로필 저장 → 분석 생성 → 결과 조회”를 실제 API로 검증

## 판단

**지금은 “기능 구현 완료”에 가깝지, “서비스 런칭 준비 완료” 상태는 아닙니다.**

특히 아래 3개는 런칭 전 필수입니다.
- 프론트 `localhost` 하드코딩 제거
- 운영 데이터 시딩/DB 준비
- LLM mock 성공 경로 제거

이 3개가 해결되지 않으면 배포 직후 바로 장애나 허위 결과가 발생할 가능성이 높습니다.
